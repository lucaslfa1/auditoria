# v1.3.146 — Remove o módulo de Dashboard/BI in-app (frontend)

**Data:** 2026-06-12
**Tipo:** Remoção de feature (decisão de produto)
**Custo:** nenhum
**Branch:** `main`

Decisão: "o módulo BI será removido do projeto; não usaremos dash por enquanto."
Confirmado escopo = **Dashboard in-app** (NÃO o export do Fechamento p/ BI
externo, que continua).

## O que foi removido (frontend)

- `src/features/dashboard/components/Dashboard.tsx` (view `dashboard`,
  "Dashboard de Qualidade").
- `src/features/dashboard/components/PerformanceDashboard.tsx` (view
  `performance` — já era **código morto**: nenhum `setView('performance')`,
  sem entrada de navegação).
- `src/App.tsx`: lazy-imports dos dois componentes + os blocos de render das
  views `dashboard` e `performance`.

A entrada do menu já havia sido removida pelo time de frontend (commit
`3ddc75cb`). O `view` default é `automacao`; uma URL avulsa `/dashboard` cai no
fallback (não quebra).

## O que NÃO foi tocado (de propósito)

- **`saveToDashboard`** (`useTranscription`/`AuditWorkspace`): apesar do nome, é
  a função de **salvar auditoria** — core, mantida.
- **`/api/dashboard/*`** (em `routers/system.py`): store de auditorias salvas
  (stats/history/save/...), usado por SavedFiles/AuditModal/PendingDispatch.
  Core — mantido.
- **Fechamento** e seu export p/ BI externo: intocados.

## Backend (analytics) — pendente de confirmação

O router `/api/analytics` (`backend/routers/analytics.py`) tinha como ÚNICOS
consumidores frontend o Dashboard/PerformanceDashboard (agora removidos) — logo
seus endpoints in-app (`indicators*`, `trend`, `top-failures`,
`matching-quality`) ficaram **sem consumidor**. PORÉM o mesmo router expõe
endpoints **externos** `/external/powerbi/*` (integração PowerBI), que não são
chamados pelo frontend e podem ter consumidor externo. Por isso a remoção do
backend foi deixada para confirmação (não derrubar PowerBI sem aval).

## Validação

- `npx tsc --noEmit`: OK.
