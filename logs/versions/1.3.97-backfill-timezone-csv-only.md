# v1.3.97 — Backfill -3h em itens CSV-only da fila com bug pre-v1.3.93

**Data:** 2026-05-28
**Tipo:** Backfill de dados (nao toca codigo)
**Custo:** zero (UPDATE 1x no Neon)

## Contexto

A v1.3.93 corrigiu o bug em `huawei_discovery._coerce_huawei_time_ms`: CSV
manifesto do OBS Huawei envia ISO em UTC mas o codigo tratava como BRT,
adicionando +3h ao epoch armazenado. So o branch **CSV-only**
(`huawei_source='obs_contact_record'`) sofria — itens merged
(`obs_contact_record+vdn`) e VDN-only (`vdn`) ja vinham corretos pela API
VDN (epoch numerico UTC).

Pos-deploy v1.3.93, novos itens entram corretos. Mas a fila atual ainda
tinha registros antigos com `huawei_begin_time`/`huawei_end_time` +3h
adiantados — afetando `audit_date` derivado, dashboards de turno/escala
e qualquer comparacao temporal por chamada.

## Diagnostico

Distribuicao por `huawei_source` em `fila_revisao_classificacao` (Neon):

| huawei_source              | total | situacao                              |
|----------------------------|-------|---------------------------------------|
| `obs_contact_record`       | 12    | 8 com diff ~3h (bug), 4 com diff ~0   |
| `obs_contact_record+vdn`   | 3     | corretos (VDN merged)                 |
| `vdn`                      | 1     | corretos                              |

Heuristica para identificar bug: `(huawei_begin_time - call_id_epoch*1000)`
entre 10_000_000 ms e 11_000_000 ms (~3h). Itens VDN/merged dao diff ~60-130s
(epoch do call_id eh o head do evento, alguns segundos antes do begin real).

Os 4 CSV-only com diff ~0 foram baixados apos o deploy v1.3.93 e ja estao
corretos.

## Mudancas

### Backfill no Neon (`auditoria-nstech-2`)

```sql
WITH csv_items AS (
  SELECT id,
    ((metadata_json::jsonb)->>'huawei_begin_time')::bigint AS begin_ms,
    ((metadata_json::jsonb)->>'huawei_end_time')::bigint AS end_ms,
    CASE WHEN split_part((metadata_json::jsonb)->>'huawei_call_id', '-', 1) ~ '^[0-9]+$'
      THEN split_part((metadata_json::jsonb)->>'huawei_call_id', '-', 1)::bigint * 1000
      ELSE NULL END AS call_id_epoch_ms
  FROM fila_revisao_classificacao
  WHERE (metadata_json::jsonb)->>'huawei_source' = 'obs_contact_record'
    AND (metadata_json::jsonb)->>'huawei_begin_time' IS NOT NULL
    AND (metadata_json::jsonb)->>'huawei_call_id' IS NOT NULL
    AND COALESCE((metadata_json::jsonb)->>'fix_timezone_applied', 'false') <> 'true'
), targets AS (
  SELECT id, begin_ms, end_ms FROM csv_items
  WHERE call_id_epoch_ms IS NOT NULL
    AND begin_ms - call_id_epoch_ms BETWEEN 10000000 AND 11000000
)
UPDATE fila_revisao_classificacao f
SET metadata_json = (
  jsonb_set(
    jsonb_set(
      jsonb_set(f.metadata_json::jsonb,
        '{huawei_begin_time}', to_jsonb(t.begin_ms - 10800000), true
      ),
      '{huawei_end_time}',
      CASE WHEN t.end_ms IS NOT NULL THEN to_jsonb(t.end_ms - 10800000)
           ELSE f.metadata_json::jsonb->'huawei_end_time' END,
      true
    ),
    '{fix_timezone_applied}', 'true'::jsonb, true
  )
)::text,
atualizado_em = NOW()
FROM targets t
WHERE f.id = t.id;
```

8 linhas afetadas (ids: 33, 37, 40, 48, 53, 54, 55, 56). Todas viraram
diff ~0.02-0.04h apos o UPDATE. Marcador `fix_timezone_applied=true` no
metadata evita re-aplicar se o backfill rodar duas vezes.

## Validacao

- ✅ Query pre-UPDATE: 8 itens com diff_hours entre 3.01 e 3.04.
- ✅ UPDATE retornou 8 linhas; todas com `fix_applied='true'` e novos
  `begin_utc` consistentes com call_id_epoch.
- ✅ Query pos-UPDATE: 12 CSV-only items, todos com diff_hours <= 0.04.
- ✅ Cross-check tabela `audits`: nenhuma auditoria com `input_hash`
  entre os 8 corrigidos — nao foram auditadas ainda, nao ha cascade fix.

## Pendencia residual (nao bloqueante)

Existem **~7 auditorias** em `audits` com `timestamp` em formato
`YYYY-MM-DDTHH:MM:SS-03:00` (BRT explicito, vindas da telefonia via
`_apply_queue_audio_timestamp`) cujas filas correspondentes ja foram
deletadas — impossivel saber via JOIN se tinham `huawei_source='obs_contact_record'`
e portanto se o `timestamp` esta +3h.

Identificadas pelo padrao no SELECT abaixo (auditorias com TZ explicito,
nao confundir com manuais que tem timestamp puro sem TZ):

```sql
SELECT id, input_hash, timestamp, audit_date FROM audits
WHERE timestamp LIKE '%-03:00' AND audit_date IS NOT NULL
ORDER BY id;
```

Decisao: nao tocar em `audits.timestamp` sem ground truth do `huawei_source`
original. Lucas pode revisar manualmente se algum relatorio mostrar horario
suspeito (>= 3h depois do real esperado).
