# v1.3.81 — Religa gate audio_inviavel + limpa setdefaults enganosos

**Data:** 2026-05-27
**Tipo:** Fix (qualidade pré-transcrição)
**Custo:** ~100-300ms por audit (pydub local no `QualityAnalyzer`)
**Bug relacionado:** BUG-031 (gate `audio_inviavel` inerte + `audio_quality.score=1.0` falso positivo)
**Memória relacionada:** `project_judge_tiebreak_2026_05_26.md` (pendência derivada da v1.3.80)

## Contexto

Investigação iniciada após a v1.3.80 (judge LLM) deixar como pendência a anomalia `audio_quality.quality="desconhecida"` e `audio_quality_score=null` no audit #10. Diagnóstico revelou dois bugs estruturais:

**Bug 1 — Gate `audio_inviavel` inerte.** O `select_transcription_candidate` em `core/transcription_selector.py:125` tem um gate de rejeição quando `audio_quality_score < min_audio_quality_score` (default 0.30). Mas em `core/transcription.py:1251`, a chamada `select_transcription_candidate(...)` **não passava** `audio_quality_score=` — o parâmetro tem default `None`, então o `if audio_quality_score is not None and audio_quality_score < threshold` jamais entrava. Áudios de volume -40dB+, 90% silêncio ou sample rate <8kHz passavam livremente para a transcrição cara (GPT-4o ~58s).

**Bug 2 — Setdefaults enganosos.** `audio/diarization_quality.py:249-250` fazia:
```python
base.setdefault("score", 1.0)
base.setdefault("quality", "desconhecida")
```
A função assumia que algum upstream populasse `audio_quality.score` antes — exatamente o papel do `core/quality_analyzer.py` (QualityAnalyzer com volume/silêncio/clipping/sample rate). Mas o `QualityAnalyzer` estava **órfão**: importado do Sentinel em v1.2.1, com testes próprios, mas zero callers em runtime. Resultado: `audio_quality.score=1.0` e `audio_quality.quality="desconhecida"` apareciam no DB para todo audit, mascarando o estado real do áudio bruto. O BI lê `audio_quality.score` como métrica de qualidade → sempre 1.0 (falso positivo permanente).

Evidência do audit #10 no Neon:
```json
{
  "score": 1.0,                          // ← default falso
  "quality": "desconhecida",             // ← default falso
  "diarization": {"score": 0.94, ...},   // ← real (pós-transcrição)
  "transcription_provider": {
    "selection_gates": {"audio_quality_score": null}  // ← gate inerte
  }
}
```

## Estratégia (A+B)

Religar o `QualityAnalyzer` no caminho hot E limpar os setdefaults. Semântica fica clara:
- `audio_quality.score` / `quality` / `notes` / `details` = **áudio bruto** (pré-transcrição, calculado pelo `QualityAnalyzer`)
- `audio_quality.diarization.score` / `quality` = **pós-transcrição** (mantido como está)

## Mudanças

### 1. `backend/core/audit.py`
- Import: `from core.quality_analyzer import QualityAnalyzer`.
- Novo helper privado `_analyze_raw_audio_quality(audio_file: bytes) -> dict`: envolve `QualityAnalyzer().analyze(audio_file)` com try/except. Degrada para `{"score": None, "quality": "desconhecida", "notes": [...], "details": {}}` em caso de falha (pydub ausente, áudio corrompido) — pipeline não quebra.
- Em `process_audit_with_ai`, antes de `transcribe_audio`:
  - `raw_audio_quality = _analyze_raw_audio_quality(audio_file)`
  - Passa `audio_quality_score=raw_audio_score` para `transcribe_audio`.
- Após `transcribe_audio`, ao montar `audio_quality` via `build_diarization_quality`, popular `score`/`quality`/`notes`/`details` com os valores do `raw_audio_quality`.

### 2. `backend/core/transcription.py`
- `transcribe_audio` ganha kwarg `audio_quality_score: Optional[float] = None`.
- `build_selector_result` (criada na v1.3.80) passa esse valor para `select_transcription_candidate(audio_quality_score=audio_quality_score, ...)`.

### 3. `backend/audio/diarization_quality.py`
- Removidos `base.setdefault("score", 1.0)` e `base.setdefault("quality", "desconhecida")`.
- Comentário explicando a divisão semântica (top-level = áudio bruto; `diarization.*` = pós-transcrição).
- Outros callers (`transcription.py:406` `_build_candidate_diarization`, `reevaluate_audit` em `audit.py:358`, testes em `test_core_logic.py`) só leem `quality["diarization"][...]` — não afetados.

### 4. `backend/tests/test_audio_quality_gate.py` (novo)
- `TestBuildDiarizationQualityNoDefaults`: 3 testes garantindo que score/quality top-level só existem quando o upstream popula.
- `TestAuditAnalyzeRawAudioQuality`: 2 testes do wrapper `_analyze_raw_audio_quality` (mock de sucesso e mock de falha).

### 5. `backend/tests/test_transcription_candidate_sandbox.py`
- 3 testes novos para o gate `audio_inviavel`:
  - `audio_quality_score=0.20` → `REJECTED("audio_inviavel")`.
  - `audio_quality_score=0.50` → `ACCEPTED("accept_fast")`.
  - `audio_quality_score=None` → gate ignorado (skip gracioso, mantém compat com chamadas legadas).

## Resultado

- **Gate `audio_inviavel` agora dispara em runtime.** Áudios com score < 0.30 (volume ≤ -40dB, ≥90% silêncio, sample rate < 4kHz) são rejeitados antes da chamada cara ao GPT-4o.
- **`audio_quality.score` no DB passa a refletir o áudio bruto.** BI e Dashboard ganham métrica útil em vez de 1.0 constante.
- **Falha do `QualityAnalyzer` é não-fatal**: log + `score=None` + pipeline segue (gate desligado naquele audit, mas não quebra a auditoria).
- **Custo extra**: ~100-300ms por audit (pydub roda em memória sobre os bytes do áudio). Negligível comparado ao tempo de transcrição (10-60s) e à economia quando um áudio inviável é rejeitado.

## Validação

- ✅ `python -m unittest tests.test_audio_quality_gate tests.test_transcription_candidate_sandbox tests.test_transcription_judge -v` — **23/23 testes** (5 audio_quality + 9 selector + 9 judge).
- ✅ Sintaxe de `core/audit.py`, `core/transcription.py`, `audio/diarization_quality.py` validada via `ast.parse`.
- ✅ Os 5 testes existentes de `build_diarization_quality` em `test_core_logic.py` continuam passando (não dependem dos setdefaults removidos).

## Pendências relacionadas

- 🟡 Threshold 0.30 do gate é conservador (só rejeita áudio extremo). Ao acumular ~50 audits com `audio_quality.score` real no DB, considerar elevar para 0.45–0.50 se a distribuição mostrar margem.
- 🟡 `QualityAnalyzer` ainda não considera codec/bitrate de telefonia G.729 explicitamente — só estima via `len(bytes) / duration_sec`. Pode dar falso positivo em áudios PCM (16kHz mono) e falso negativo em áudios super-comprimidos. Aceitável por enquanto.
- 🟡 UI/Dashboard precisa começar a exibir `audio_quality.notes` ao auditor (informações como "Volume muito baixo", "Codec muito comprimido (~8kbps) - típico de telefonia"). Hoje só `quality` é mostrado.
