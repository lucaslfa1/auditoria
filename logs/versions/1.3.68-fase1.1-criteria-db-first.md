## Objetivo
Inverter a direcao do catalogo de criterios: `classification.py` passa a ler de
`audit_sectors` + `audit_alerts` em runtime em vez de `scoring_rules.yaml`. Edicoes
via UI agora deixam trail em `audit_*_audit_log` e — combinado com a Fase 1.2 — se
tornam permanentes.

## Contexto
Antes desta versao, o `load_audit_criteria_catalog` em `classification.py:162` lia
exclusivamente do YAML. Qualquer mudanca via UI no admin (CRUDs ja existentes em
`repositories/admin_criteria.py`) gravava no DB mas era ignorada pela IA — e
sobrescrita no proximo deploy pelo seed UPSERT. **Causa raiz da percepcao de
"sistema engessado".**

## Mudancas

### Banco (Neon `auditoria-nstech-2`, ja aplicado em producao)

Migration `m20260515_002_audit_criteria_audit_log` (idempotente):

1. **`pop_ref TEXT`** em `audit_alerts` (aditiva, nullable) + index parcial em
   `pop_ref` para lookups por POP id (e.g. "4.1.5").

2. **Tres audit_log tables** (`audit_sectors_audit_log`, `audit_alerts_audit_log`,
   `audit_criteria_audit_log`) com schema unificado:
   ```sql
   id, acao ('create'|'update'|'delete'), entity_id,
   payload_antes JSONB, payload_depois JSONB,
   alterado_por, alterado_em, motivo,
   origem ('ui'|'api'|'seed'|'script'|'system'|'migration')
   ```
   Indexes em `(entity_id, alterado_em DESC)` e `(alterado_em DESC)`.

3. **Backfill manual de `pop_ref`** (via MCP) para os 77 alertas existentes — mesmo
   mapping que o YAML usa (4.1.x risk, 4.2.1 cadastro, 4.3.x unilever, 4.4.x
   logistica, 4.5.x mondelez, 4.6.x checklist, 4.7.x receptivo).

### `backend/repositories/admin_criteria.py`
- 9 funcoes CRUD (`create/update/delete` x `sector/alert/criterion`) ganham
  `alterado_por` **obrigatorio** + `motivo` + `origem` (padrao Fase 0).
- Cada mutacao captura snapshot anterior via SELECT e grava em `*_audit_log` na
  **mesma transacao**.
- Updates no-op (mesmo payload reaplicado) NAO geram linha de audit_log — evita
  poluir trail.
- `pop_ref` opcional em `create_alert`/`update_alert` (preserva valor antigo se
  nao informado).
- Novo `list_audit_log(get_connection, *, entity_type, entity_id=None, limit=50)`.

### `backend/routers/admin_criteria.py`
- Pydantic models (`SectorCreate/Update`, `AlertCreate/Update`, `CriterionCreate/
  Update`) ganham campo `motivo: Optional[str]`. `AlertCreate/Update` ganham
  `pop_ref: Optional[str]`.
- 9 endpoints de mutate propagam `alterado_por=user.username` + `motivo`.
  Delete endpoints aceitam `motivo` como query param.
- **Novo endpoint** `GET /api/admin/criteria/audit-log?entity_type=sector|alert|
  criterion&entity_id=...&limit=...` (admin-only).
- **Novo endpoint** `POST /api/admin/criteria/cache/invalidate` (admin-only) que
  invalida o `lru_cache` do catalogo. Tambem chamado automaticamente apos cada
  mutacao bem-sucedida — admin nao precisa lembrar de invalidar.

### `backend/classification.py`
- `load_audit_criteria_catalog` agora le do DB por padrao
  (`_load_catalog_from_db`), mantendo o `@lru_cache(maxsize=1)`.
- Logica de replicacao BAS→sibling sectors movida para helper compartilhado
  `_apply_operational_siblings(catalog, sector_labels)` — mesma semantica de antes
  (transferencia, distribuicao, fenix, bbm herdam alertas BAS).
- **Fallback de seguranca:** env `CRITERIA_CATALOG_SOURCE=yaml` reverte para
  leitura YAML sem deploy (use em incidentes).
- Excecao no caminho DB tambem cai de volta no YAML automaticamente.

### `backend/db/scoring_loader.py` e `backend/database.py`
- `build_seed_data_from_yaml` agora retorna tupla `(alert_id, sector_id, label,
  context, pop_ref, criteria_tuples)` (insercao do `pop_ref`).
- `_seed_audit_criteria` em `database.py` propaga `pop_ref` no UPSERT de alertas.
  **Continua rodando neste PR** — fica como rede de seguranca. Sera neutralizado
  na Fase 1.2 (1.3.69).

### `backend/scripts/validate_catalog_parity.py` (NOVO)
- CLI: `python -m scripts.validate_catalog_parity`.
- Carrega catalogo via YAML e via DB, faz diff sector/alert/pop_ref, sai com
  exit code 1 se houver qualquer diferenca.
- Executado contra o estado de producao apos a aplicacao desta versao:
  **12 setores, 77 alertas, zero diff**.

## Comportamento Pos-Fix
- IA usa o DB como fonte do catalogo. Edicoes admin se refletem no `lru_cache`
  apos a invalidacao automatica (ou via `POST /api/admin/criteria/cache/invalidate`).
- Toda mutacao admin deixa linha em `audit_*_audit_log` com `payload_antes/depois`
  em JSONB — UI pode mostrar diff "antes vs depois" sem schema rigido.
- Para o auditor mestre: pode editar peso/label/descricao de criterio pela UI e ja
  ver a mudanca refletida na proxima triagem (sem PR + deploy).

## Riscos
- **Cache lru per-process:** invalidacao so afeta o pod que recebeu a request.
  Em Cloud Run multi-pod, pods adicionais usam catalogo cacheado ate proximo
  restart. Catalogo e pequeno (77 alertas) — alternativa sem cache merece consideracao
  em fase futura.
- **Mudanca de assinatura nos repositorios:** todos os call sites locais foram
  atualizados, mas qualquer script externo que importe `create_sector` etc. sem
  passar `alterado_por` vai retornar `False`+log ERROR (visivel, nao silencioso).
