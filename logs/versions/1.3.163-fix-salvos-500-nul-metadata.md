# v1.3.163 — Fix 500 em /api/salvos (metadata_json com escape de NUL U+0000)

**Data:** 2026-06-15
**Tipo:** Correção de bug (incidente de produção)
**Custo:** nenhum
**Branch:** `main`

## Incidente

`GET /api/salvos?tipo=auditoria` retornava **500** em produção (toast "Erro no
Servidor / Request failed (500)" na tela Arquivos Salvos). Diagnóstico via
`gcloud logging` (request log) + reprodução da query contra o banco de prod
(read-only, Neon MCP).

## Causa raiz

`list_arquivos_salvos` casta `metadata_json::jsonb` numa subquery LATERAL. Uma
linha (`arquivos_salvos.id=206`, audit 206) tinha o escape de NUL (U+0000, a
sequência barra-invertida + `u0000`) no `metadata_json`. Esse escape é JSON
**válido a nível de texto** (por isso `IS JSON` aceita), mas o **`jsonb` do
Postgres o REJEITA** (`unsupported Unicode escape sequence`). Como o cast é numa
LATERAL sobre a tabela toda, **UMA linha corrompida abortava a query inteira** → a
lista inteira caía em 500. Local não reproduzia (banco de teste sem essa linha).

## O que foi feito

1. **Dado (desbloqueio imediato, prod):** `UPDATE arquivos_salvos SET
   metadata_json = replace(metadata_json, <escape NUL>, '') WHERE id = 206` (alvo
   único, idempotente, verificado antes). Removeu 29 ocorrências do escape
   (14016→13842 chars); a query voltou a rodar (193 auditorias listáveis, 0 linhas
   ruins restantes). Prod desbloqueada sem deploy.
2. **Código (resiliência, este commit):** o cast na LATERAL agora remove o escape
   antes de castar — `ELSE replace(a.metadata_json, chr(92) || 'u0000', '')::jsonb`
   (`chr(92) || 'u0000'` constrói a sequência do escape de NUL sem usar literal de
   barra invertida no código). Assim, uma futura linha corrompida não derruba mais
   a lista — o escape é ignorado e o resto do metadata é lido normalmente.
3. **Regressão:** `tests/backend/test_saved_files_nul_metadata.py` insere uma linha
   com o escape de NUL e exige que `list_arquivos_salvos` NÃO levante e devolva o
   metadata sanitizado.

## Validação

- Repro local do fix (banco de teste): linha com o escape presente → lista OK,
  metadata sanitizado (`antesdepois`).
- Novo teste de regressão: passa.
- Suite backend completa (banco de teste; nunca prod): verde.

## Pendências relacionadas (não neste commit)

- **Origem do dado ruim:** algum caminho de escrita gravou o NUL no metadata
  (provável NUL numa transcrição/summary). Sanitizar na ESCRITA evitaria a raiz —
  vale um passo futuro.
- Outros pontos que castam `metadata_json::jsonb` (ex.: fechamento/dashboard)
  podem ter a mesma fragilidade; o dado de prod já foi limpo (0 linhas com NUL),
  mas convém aplicar o mesmo `replace` defensivo se aparecerem.
