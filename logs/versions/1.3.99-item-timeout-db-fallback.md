# v1.3.99 — Item timeout passa a aceitar fallback DB (corrige env de 70s zumbi)

**Data:** 2026-05-28
**Tipo:** Bug — automacao
**Custo:** deploy backend (mudanca de codigo) + 1 config no Neon (ja aplicada)

## Contexto

Apos a v1.3.98 aplicar `automacao_audit_time_budget_seconds=540` e
`automacao_audit_batch_size=10` no DB, o ciclo 784 (16:15Z) ainda retornou
erro:

```
Timeout ao auditar 'ligacao_huawei_loren_kelly_freitas_pinheiro_*.wav'
apos 69.99344485799975s. O item foi abortado para evitar automacao zumbi.
```

70s nao bate com nenhum dos valores configurados (480 default, 540 cap,
600 transcription). Investigacao confirmou:

`backend/core/automation.py:289` lia **so** de env:
```python
def _get_automation_item_timeout_seconds() -> int:
    raw = os.getenv("AUTOMATION_ITEM_TIMEOUT_SECONDS", "480")
```

Diferente de `batch_size` e `time_budget` que tem fallback DB, esse cai
direto no default 480 se env vazio. Mas se env esta setado em outro valor
(suspeita: Gemini setou `=70` em algum teste anterior), nao havia como
sobrescrever via config — o env sempre ganhava silenciosamente.

Resultado: 2 audits passaram no ciclo 784 (Caio #43, Asdrubal #44 — casos
rapidos), o restante (Loren, Thais, Raffaella) foi morto antes do Azure
Fast Transcription terminar.

## Mudancas

### `backend/core/automation.py`

`_get_automation_item_timeout_seconds` agora padronizado com
`_get_automation_audit_batch_size`:

```python
def _get_automation_item_timeout_seconds() -> int:
    raw = os.getenv("AUTOMATION_ITEM_TIMEOUT_SECONDS")
    if raw in (None, ""):
        raw = database.get_config_value("automacao_item_timeout_seconds", "480")
    try:
        parsed = int(str(raw).strip())
    except (TypeError, ValueError):
        parsed = 480
    return max(60, min(parsed, 540))
```

Comportamento:
- Env setado e nao-vazio -> usa env.
- Env vazio ou ausente -> usa DB (`automacao_item_timeout_seconds`).
- Cap 60..540s mantido.

### `tests/backend/test_automation_item_timeout.py` (novo)

6 testes RED->GREEN:
- `test_env_set_wins` — env=300 ganha, DB nao chamado.
- `test_env_empty_falls_back_to_db` — env="" -> usa DB=540.
- `test_env_missing_falls_back_to_db` — sem env -> usa DB=540.
- `test_env_missing_and_db_empty_uses_default_480` — sem env + sem DB -> 480.
- `test_cap_max_540` — env=9999 -> 540.
- `test_cap_min_60` — env=10 -> 60.

### Configuracao no Neon (`auditoria-nstech-2`) — ja aplicada

```sql
INSERT INTO configuracoes (chave, valor, descricao, tipo, is_secret, atualizado_em)
VALUES ('automacao_item_timeout_seconds', '480', '...', 'int', false, NOW())
ON CONFLICT (chave) DO UPDATE SET valor = EXCLUDED.valor, atualizado_em = NOW();
```
