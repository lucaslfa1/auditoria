## Objetivo
Consolidar os 3 dicionarios + 2 ladders hardcoded de mapeamento setor cru →
canonico em uma unica tabela `sector_aliases`, editavel via UI sem deploy.
Fechamento da Fase 2 do plano DB-first.

## Contexto
Antes desta versao, qualquer ajuste em mapeamento de setor (ex: nova escala
"RJ-VERDE" que precisa cair em UTI, ou supervisor novo que assume Longo Percurso)
exigia PR + deploy + 5 min de CI. As regras viviam em:

- `backend/classification.py:_resolve_db_sector_alias` (`sector_alias_map`, 12 entries)
- `backend/repositories/operators.py:_SECTOR_ALIASES` (9 entries)
- `backend/repositories/operators.py:map_db_sector_to_classification_sector`
  (ladder if/elif com supervisor/setor/escala — ~50 linhas)
- `backend/repositories/operators.py:_map_organizacao_telefonia_to_sector`
  (ladder if/elif para `organizacao_telefonia` — ~25 linhas)

## Comportamento Pos-Fix
- Auditor mestre pode cadastrar nova regra (ex: "supervisor_contains 'maria
  silva' → distribuicao") via UI e a IA / fluxo de operadores usa imediatamente
  na proxima requisicao (cache invalidado automaticamente).
- Toda mutacao gera linha em `sector_aliases_audit_log` com payload JSONB
  antes/depois — historico completo de quem mexeu em qual regra.
- O comportamento de mapeamento e **byte-a-byte identico** ao codigo
  pre-1.3.71, validado pelo script de parity.

## Riscos
- **lru_cache per-process (inerdado da Fase 1.1)**: invalidacao so afeta o pod
  que recebeu a request. Cloud Run multi-pod: pods adicionais usam catalogo
  cacheado ate proxima rotacao.
- **Divergencia historica `celula_atendimento` (codigo) vs `receptivo`
  (audit_sectors)**: preservada nesta migracao. Regras que mapeiam para
  `celula_atendimento` continuam apontando para um id que **nao existe** em
  `audit_sectors`. Saneamento aceito como dividas futura, fora do escopo
  desta fase.
- **Sem FK em `canonical_sector_id`**: nao bloqueia INSERT com sector_id
  inexistente. Validacao fica do lado do auditor — pode causar
  "regra orfa" que nao casa com setor de IA. Tradeoff: permite criar setores
  em ordem qualquer (regra antes de setor ou vice-versa).
- **Frontend ainda nao tem UI para sector_aliases**: backend pronto, mas o
  auditor mestre por enquanto so consegue editar via curl/Postman. UI virá
  numa Fase 2.1 separada quando alguem pedir.
