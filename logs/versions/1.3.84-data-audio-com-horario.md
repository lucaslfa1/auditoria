# v1.3.84 — Mostrar horário do áudio junto da data (Arquivos Salvos + AuditModal)

**Data:** 2026-05-27
**Tipo:** UX/Frontend
**Custo:** zero
**Issue:** Lucas pediu "em data do áudio também informe o horário do audio".

## Contexto

Até v1.3.83, o card "Data do áudio" nos painéis de Arquivos Salvos (Automação) e AuditModal exibia apenas a data (`26/05/2026`). Para ligações vindas do pipeline Huawei, o backend já persiste `audits.timestamp` com o horário real da chamada (`beginTime` em São Paulo TZ), mas o frontend ignorava esse campo no display.

Snippet da fonte (`backend/core/automation.py:969-977`):
```python
if "huawei_begin_time" in metadata:
    dt = datetime.fromtimestamp(int(metadata["huawei_begin_time"]) / 1000.0, tz=timezone.utc)
    br_dt = dt.astimezone(ZoneInfo("America/Sao_Paulo"))
    real_timestamp = br_dt.isoformat()
    if not audio_date:
        audio_date = br_dt.strftime("%Y-%m-%d")
```

Resultado: `audits.timestamp` já carrega o horário real da chamada para audits Huawei (a maioria do volume). Para uploads manuais, `timestamp` cai para o instante de criação do audit — pode não casar com `audio_date`.

## Estratégia

Helper único `formatAudioMoment(audioDate, timestamp)` decide:
- Se `timestamp` existe E a data dele (em São Paulo TZ) bate com `audioDate` → exibe `dd/mm/yyyy HH:mm`.
- Se só `audioDate` existe ou dates divergem → exibe `dd/mm/yyyy` (mantém comportamento antigo).
- Se só `timestamp` existe → exibe `dd/mm/yyyy HH:mm`.
- Se nada → string vazia (card nem renderiza).

Heurística "datas batem → horário é confiável" protege uploads manuais cujo `timestamp` é instante de criação (não da ligação) mas, por coincidência, está no mesmo dia. Trade-off: para esses casos, mostra horário não-100%-verdadeiro. Aceitável porque (a) volume é dominado pelo pipeline Huawei e (b) divergência diária é fallback seguro.
