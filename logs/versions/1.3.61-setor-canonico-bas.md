## Objetivo
- Garantir que sub-escalas de BAS/UTI (cor/RJ/COMBO) sejam aglutinadas no setor canonico (`bas`/`uti`) no fluxo de classificacao e auditoria.
- Preservar a granularidade nas tabelas `colaboradores.setor` e `colaboradores.escala` para o fechamento, que tem calibracao propria.

## Contexto
`operadores_seed.json` armazena setores com sub-escalas literais (ex: `"BAS - AZUL"`, `"UTI - VERDE"`, `"BASE PR - Amarela"`, `"BASE DE PR - CINZA"`). O catalogo de criterios de auditoria conhece apenas os ids canonicos `bas`, `uti`, etc. — sem variantes por cor.

O fechamento (`backend/core/fechamento_service.py:50`) ja faz sua propria normalizacao com `_normalize_setor` que distingue `UTI (RJ)` de `UTI`, mantendo a calibracao oficial.

O caminho de classificacao/auditoria, porem, recebia o setor cru e:
- Para `UTI - AZUL` → mapeava OK (`startswith("uti")`).
- Para `BAS - AZUL`, `BASE PR - AZUL`, `BAS - CINZA` etc. → **NAO mapeava** (regra era `== "bas"`).
- Resultado: `_get_effective_db_sector` caia no fallback e devolvia a string crua `"BAS - AZUL"`, que nao bate com nenhum `sector_id` do catalogo.

## Causa Raiz
`backend/repositories/operators.py:234` em `map_db_sector_to_classification_sector`:
```python
if normalized_setor == "bas":  # so pega "BAS" exato
    return "bas"
```
E o espelho em `_matches_operador_sector` (linha 278):
```python
if normalized_sector_id == "bas":
    return mapped_setor == "bas" or normalized_setor == "bas"
```

## Mudancas

### Backend
- **`backend/repositories/operators.py:234`**: trocar `== "bas"` por `.startswith("bas")` em `map_db_sector_to_classification_sector`. Isso passa a pegar `BAS`, `BAS - AZUL/CINZA/AMARELA/VERDE`, `BASE PR`, `BASE PR - *`, `BASE DE PR - *` (porque `"base"` comeca com `"bas"`).
- **`backend/repositories/operators.py:278`**: mesma troca em `_matches_operador_sector`, para manter o comportamento simetrico (verificacao de pertencimento ao setor).
- Adicionado comentario explicando que UTI/BAS aglutinam sub-escalas e a granularidade fica para o fechamento.

## Principio Adotado
- **Fechamento** — preserva granularidade (UTI vs UTI(RJ), BAS - AZUL vs BAS - CINZA) porque a planilha oficial pondera diferente. Le direto de `colaboradores.setor` e `colaboradores.escala`, aplica `_normalize_setor` proprio.
- **Sistema (classificacao + auditoria)** — usa apenas o id canonico (`bas`, `uti`). Sub-escala nao discrimina criterios.

## Impacto
- Ligacoes de operadores cadastrados como `BAS - AZUL` etc. passarao a ter `db_sector="bas"` no prompt da IA e nos guardrails — antes vinham com string crua e ficavam orfas do catalogo.
- Auditorias destes operadores passam a casar com os criterios do setor `bas`.
- Fechamento **inalterado**.

## Riscos
- Baixo: `.startswith("bas")` engloba todas as variantes vistas no seed sem colisao (nenhum outro setor canonico comeca com "bas").
