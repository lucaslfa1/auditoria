# v1.3.115 — Direção via consulta VDN substitui a pré-triagem por áudio (Whisper+GPT)

**Data:** 2026-06-11
**Tipo:** Otimização de custo — recomendação 5.3 do relatório de consumo Azure (10/06)
**Custo:** deploy backend; corta 2 chamadas Azure pagas por candidata OUTBOUND de setor de risco
**Branch:** `handover`

## Contexto

Para setores de risco OUTBOUND-only (uti, bas, distribuicao, fenix,
transferencia), após o download o sync rodava `analyze_call_direction`
(`core/pre_triage.py`): 30s de áudio → GPT-4o diarize + GPT-4o chat = **2
chamadas pagas por candidata**, para confirmar se a ligação era ativa ou
receptiva. O relatório de consumo recomendou substituir pela consulta de
direção via VDN (`consultar_direcao_chamada`, huawei_client.py), que já
existia mas nunca foi integrada ao gate.

## Mudança (`backend/core/huawei_sync.py`, `_processar_candidato`)

Ordem de resolução da direção (sem custo Azure):
1. **Consulta VDN por callId** (`client.consultar_direcao_chamada`) — evidência
   real da Huawei. Vem PRIMEIRO porque o `isCallIn` dos metadados pode ser
   rótulo sintético derivado da direção da query (a mentira que a pré-triagem
   por áudio existia para pegar). Coerção estrita: só `True`/`False` contam.
2. **Metadados da interação** (`_resolve_huawei_is_call_in`) — fallback quando
   a VDN falha/não responde (API ainda não validada em produção; fail-open
   evita starvation dos setores de risco).

Semântica preservada (contrato `audio_direction_pre_triage` intacto):
- receptiva → `inbound_quarantine`, descarta (failure_reason novo:
  `receptiva_direcao_vdn`);
- ativa → `outbound`, enfileira;
- indeterminada → `indeterminada_descartada` (na dúvida não audita receptiva).

Contadores mantidos (`pretriagem_direcao_*`) — consumidores de telemetria não
quebram; a fonte (`vdn_api`/`metadata`) sai no log.

## O que NÃO mudou
- `core/pre_triage.py` ainda existe (remoção na etapa de código morto,
  v1.3.120 planejada) — apenas não é mais chamado pelo sync.
- Guardrail textual EFETUADA vs RECEPTIVA (v1.3.73) segue ativo na avaliação
  como defesa downstream.
- Filtro pré-download `_should_skip_call` segue descartando direção
  desconhecida antes do download (o gate pós-download virou defesa em
  profundidade).

## Riscos e observabilidade
- A API `querydetailcallinfo`/`querybasiccallinfo` por callId está marcada
  como "pendente de validação em produção". Se ela não responder, o fluxo cai
  nos metadados (comportamento equivalente ao filtro upstream). Acompanhar
  pelos logs `fonte=vdn_api` vs `fonte=metadata` no primeiro ciclo pós-deploy.
- Possível aumento de `pretriagem_direcao_indefinida` se metadados e VDN
  falharem juntos — visível na telemetria do sync.

## Validação
- ✅ `tests/backend/test_huawei_sync.py` — 4 cenários do novo gate:
  receptiva via VDN descarta (vence isCallIn sintético), ativa via VDN
  enfileira, indeterminada descarta (defesa em profundidade), VDN indisponível
  cai para metadados. 77 passed no arquivo.
- ✅ Suite completa: 28 falhas = MESMAS do baseline (3 de
  `test_classificar_audio_huawei_*` comprovadas pré-existentes no commit base
  — dependência de ordem por catálogo de critérios incompleto no seed; vão
  para a etapa "suite 100% verde").

## Achado registrado para a etapa de testes (v1.3.119 planejada)
Seed de banco limpo (`_seed_audit_criteria`) cobre só `logistica` (1 setor,
4 alertas, 48 critérios). O catálogo completo de produção (uti, bas, fenix,
distribuicao, transferencia...) foi importado via UI/scripts e não tem seed.
Testes de guardrail de classificação dependem dele — por isso falham em DB
recém-criado.
