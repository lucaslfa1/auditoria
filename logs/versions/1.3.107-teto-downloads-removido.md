# v1.3.107 - Remove teto de UI do "Limite de downloads" (automação)

**Data:** 2026-06-03
**Tipo:** Fix (UX/config) — limite de downloads volta a ser 100% definido pela configuração
**Status:** implementado, `tsc -b` limpo; aguardando deploy (CI/CD)

## Contexto

Lucas notou que o "Limite de downloads" por ciclo (`huawei_d1_limite_ligacoes`)
estava travado em no **máximo 100**, sendo que o limite deveria ser definido só pela
configuração.

Diagnóstico: o limite **sempre foi config-driven** — `huawei_d1_limite_ligacoes`
(lido em `backend/core/huawei_sync.py:2134`, só com piso de 1, default 20, **sem teto**).
O "máximo 100" era **clamp de frontend**, em dois pontos:
- `src/features/automacao/hooks/useAutomacaoDashboard.ts` — `FIELD_LIMITS.limite_ligacoes = { min: 1, max: 100 }`
  + `Math.min(limits.max, ...)` cortava qualquer valor >100 **antes de gravar**.
- `src/features/automacao/components/AutomationConfigPanel.tsx` — `<NumberInput max={100} />`.

O save (`POST /api/configuracoes` → `update_config`) e o read (summary D-1) **não**
fazem clamp. Ou seja, o teto era puramente cosmético/UI.

## Mudança

- `useAutomacaoDashboard.ts`: `FIELD_LIMITS` agora aceita `max` opcional; `limite_ligacoes`
  passa a `{ min: 1 }` (sem teto); sanitização usa `Math.min(limits.max ?? Infinity, ...)`.
- `AutomationConfigPanel.tsx`: prop `max` do `NumberInput` virou opcional; removido o
  `max={100}` do campo `limite_ligacoes`.

Resultado: o campo aceita qualquer valor ≥1; o valor efetivo é o que estiver salvo na
config (backend já honra sem teto). Demais campos mantêm seus limites.

## Observação (não alterado)

- `AutomationSettings.tsx` usa `'100'` apenas como **default de fallback** quando a config
  não está setada (não é um teto). Default "real" do backend é 20 — inconsistência
  cosmética antiga, deixada como está para não mexer em comportamento sem pedido.
- Para passar de 100 sem UI, o env `HUAWEI_SYNC_MAX_DOWNLOAD_ATTEMPTS` tem precedência
  sobre a config (já era assim).
