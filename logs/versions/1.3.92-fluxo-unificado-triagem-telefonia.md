# v1.3.92 — Fluxo unificado em Triagem e Telefonia + badge Auto por origem

**Data:** 2026-05-27
**Tipo:** UX + bug — itens manuais e automaticos deixam de sumir da Triagem
**Custo:** zero (deploy frontend + backend)

## Contexto

Lucas (auditoria): "não estou vendo mais os arquivos baixados chegando na triagem, pode ter ocorrido quebra... é pra deixar tudo no fluxo e eu acredito que isso impactou ligações que baixam manual também".

Decisao: fluxo unico em Triagem (auto + manual juntos). A distincao visual fica
no badge "Auto" do frontend, que olha `metadata.is_manual` + `metadata.origem`.

## Diagnostico (antes da v1.3.92)

A query da Triagem (`listar_fila_revisao_classificacao` com `status=pending`) tinha
dois filtros que escondiam itens vindos do Huawei sync:

1. **Clausula NOT em `classification_review.py:732-736`**: itens com
   `origem='huawei_sync'` + `classification_status='pending'` + `is_manual='false'`
   so apareciam depois que a fase 2 da automacao rodasse e flipasse o
   `classification_status` para `'done'`. Se a fase 2 quebrasse ou nao rodasse,
   eles ficavam invisiveis indefinidamente.
2. **`status_override = "downloaded"` em `huawei_sync.py:1423`**: sync manual
   gravava `status='downloaded'`, que nao matchava o filtro `status=pending` da
   Triagem. Manual nunca aparecia na Triagem; ficava so na Telefonia ate alguem
   clicar "Enviar para Triagem".

Sintoma para o auditor: Triagem vazia mesmo com sync rodando + manual download
"sumindo" depois de baixar.

## Decisoes

1. **Backend `classification_review.py`** — query do `status=pending` simplificada:
   - Remove a clausula `AND NOT (huawei_sync AND class_pending AND non-manual)`.
   - Inclui `status='downloaded' AND origem='huawei_sync'` como fallback para
     itens legados (manual sync antes da v1.3.92 ficou com `'downloaded'`).
2. **Backend `huawei_sync.py:1423`** — `status_override = None` sempre. Manual e
   auto entram com o mesmo status (regra padrao: `pending` se precisa_revisao,
   `auto_resolved` caso contrario).
3. **Frontend `RemoteTriageQueue.tsx`** — badge Auto agora aparece quando
   `metadata.origem === 'huawei_sync'` AND `metadata.is_manual !== true`,
   independente de `classification_status`. Manual: sem badge.
4. **Frontend `RecordingsList.tsx`** (Telefonia) — mesma logica de badge. Como o
   endpoint `/api/telefonia/recordings` ja filtra `origem='huawei_sync'`, basta
   checar `metadata.is_manual`.

## Mudancas

### `backend/repositories/classification_review.py`
- Linhas 726-753: query do filtro `status=pending` simplificada (remove o NOT
  da huawei_sync + adiciona fallback de `downloaded` para legacy).

### `backend/core/huawei_sync.py`
- Linha 1423: `status_override = None` (era `"downloaded" if is_manual else None`).

### `src/features/classifier/components/RemoteTriageQueue.tsx`
- Badge Auto criterio mudou de `classification_status === 'done'` para
  `metadata.origem === 'huawei_sync' && metadata.is_manual !== true/'true'`.

### `src/features/telefonia/components/RecordingsList.tsx`
- Badge Auto criterio mudou de `classification_status === 'done'` para
  `metadata.is_manual !== true/'true'`.

### `tests/backend/test_huawei_sync.py`
- `test_enfileirar_audio_manual_retem_em_telefonia_ate_envio` virou
  `test_enfileirar_audio_manual_marca_is_manual_em_metadata`: agora assert
  `status_override is None` + `metadata.is_manual is True`.
