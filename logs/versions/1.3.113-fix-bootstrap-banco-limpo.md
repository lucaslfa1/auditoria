# v1.3.113 — Bootstrap em banco limpo: commit por migration (fix handover)

**Data:** 2026-06-11
**Tipo:** Bug — infraestrutura de banco (descoberto na preparação do handover)
**Custo:** nenhum em prod (caminho só afeta bancos com migrations pendentes em lote)
**Branch:** `handover` (trabalho de preparação para o time de engenharia)

## Contexto

Ao provisionar um banco de teste LIMPO (PostgreSQL 16 local) para o baseline
do handover, o `init_db()` falhou em `m20260518_004_migrate_bbm_to_distribuicao`:

```
psycopg2.errors.UndefinedTable: não existe a relação "sector_aliases"
```

A tabela `sector_aliases` é criada por `m20260515_003` (anterior na ordem), mas
o runner (`db/migrations.py:run_pending_migrations`) aplicava TODAS as
migrations pendentes num único cursor/transação, com commit só no final do
`init_db()`. A `m20260518_004` escreve via `repositories.sector_aliases`, que
abre **conexão própria do pool** — e essa conexão nova não enxerga a tabela
ainda não commitada da transação do runner.

Em produção nunca quebrou porque as migrations foram aplicadas
incrementalmente (cada deploy commitava as suas). O bug só aparece em
**bootstrap de banco do zero** — exatamente o cenário do time de engenharia
ao subir o ambiente deles no Azure.

## Mudança

### `backend/db/migrations.py`

Novo `_commit_migration_step(cursor)`, chamado após cada migration aplicada:

- Cada migration vira atômica e fica visível para conexões novas abertas
  pelas migrations seguintes (padrão Alembic).
- Falha em uma migration posterior preserva o progresso já aplicado
  (marcas em `schema_migrations` commitadas junto com o step).
- Guard defensivo (`getattr(cursor, "connection", None)`) mantém compatível
  com cursores fake de testes.
