## Objetivo
Implantar a infraestrutura de auditoria para `configuracoes` (Fase 0 do plano de migracao
DB-first). Pre-requisito para qualquer config dinamica futura: toda mudanca em runtime
passa a deixar trail de quem/quando/por que.

## Contexto
O documento `docs/database/dynamic-config-migration.md` (criado nesta mesma sessao)
mapeou tudo que ainda esta engessado em arquivo/hardcoded e propos um plano em 7 fases.
Esta versao executa a **Fase 0 (PRE-REQUISITO)**: sem audit_log nas tabelas vivas,
qualquer mudanca em runtime fica invisivel — ja tivemos incidentes onde "alguem mexeu
no prompt" sem trail.

## Comportamento Pos-Fix

- Toda mudanca em `configuracoes` deixa linha em `configuracoes_audit_log` com
  `valor_antes / valor_depois / alterado_por / motivo / origem / alterado_em`.
- UPSERT que nao muda valor (mesmo valor reaplicado) NAO gera linha de audit — evita
  poluicao.
- GET `/api/configuracoes` mascara os 6 secrets — UI agora mostra `***` em vez de
  vazar AK/SK Huawei.
- GET `/api/configuracoes/audit-log` permite ao admin ver "quem mexeu na config X".
- Convencao de `alterado_por`:
  - `<username>` (ex.: `Lucas`) — vindo do router admin (origem=ui).
  - `system:<modulo>` (ex.: `system:automation`, `system:automation_engine`) — chamadas
    internas autonomas (origem=system).
  - `script:<nome>` (ex.: `script:update_config_prompt`) — scripts manuais
    (origem=script).
  - `seed`/`migration` — reservados para bootstrap/migrations futuras.

## Riscos
- **Breaking change na API interna de `update_config`**: chamadas sem `alterado_por`
  passam a retornar `False` + log ERROR. Todos os call sites do repositorio foram
  atualizados nesta versao; nao ha chamada legada conhecida. Se aparecer alguma 3rd
  party (script externo, notebook), vai quebrar barulhento (visivel no log) — nao
  silencioso.
- **Mascaramento de secrets na UI**: o GET passa a devolver `***` para os 6 secrets.
  O risco de round-trip (UI enviar `***` de volta no save e corromper o segredo) foi
  mitigado no proprio repositorio: o `update_config` detecta `valor='***' AND is_secret=true`
  e retorna no-op + 200, preservando o segredo real. Ainda assim vale validar visualmente
  na tela `Configuracoes.tsx` que o input dos secrets nao parece editado quando deve estar
  vazio (pra evitar UX confusa).
