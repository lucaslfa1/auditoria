## Objetivo
Fechar o ciclo da Fase 1 do lado do auditor mestre. A Fase 1.1 (1.3.68) e 1.2
(1.3.69) deixaram o backend DB-first e nao destrutivo, com `audit_*_audit_log`
ja capturando `payload_antes/depois`, `alterado_por`, `motivo`. Esta versao expoe
isso na UI: o admin **precisa** informar motivo a cada mutacao e consegue ver o
historico de cada setor/alerta/criterio direto no painel.

## Contexto
Apos as Fases 1.1+1.2, o backend ja:
- Aceita `motivo` em todos os 9 endpoints CRUD (`POST/PUT/DELETE` x setor/
  alerta/criterio).
- Aceita `pop_ref` opcional em `POST/PUT /api/admin/alerts`.
- Expoe `GET /api/admin/criteria/audit-log?entity_type=&entity_id=&limit=`.

A UI nao consumia nada disso — `motivo` era sempre vazio, `pop_ref` nao tinha
campo, e o auditor mestre nao tinha como ver o historico de mudancas.

## Mudancas

### `src/features/admin/components/AdminCriteriaPage.tsx`
- **Campo `motivo` obrigatorio** em todos os modais (Setor, Alerta, Criterio).
  Render via `<MotivoField>` reutilizavel. Validacao no front via `ensureMotivo`:
  se vazio, mostra toast e bloqueia submit (sem chamar o backend).
- **Campo `pop_ref`** adicionado ao modal de Alerta + interface `Alert.pop_ref`
  + mapping em `fetchData`. Tambem exibido como badge `POP X.Y.Z` ao lado do
  label do alerta na lista.
- **Modal de delete unificado** substituindo `confirm()` nativo. Estado
  `deleteTarget: { type, id, label }` + `deleteMotivo`. Antes de chamar
  `DELETE`, valida motivo nao vazio e propaga via query param
  `?motivo=...` (compatibilidade com o contrato backend que ja aceita).
- **Drawer de historico** (slide-in lateral) por entidade. Cada linha de setor/
  alerta/criterio ganha botao `<History>` (icone clock). Ao clicar, abre o
  drawer e dispara `GET /api/admin/criteria/audit-log?entity_type=...&entity_id=...&limit=50`.
- **Renderizacao do audit_log:**
  - Cada entrada e um card com badge da acao (create=verde, update=ambar,
    delete=vermelho), timestamp formatado em pt-BR, usuario, origem, motivo.
  - Click expande pra mostrar **diff antes/depois**:
    - `create`: lista de chaves criadas em verde.
    - `delete`: lista de chaves removidas em vermelho.
    - `update`: so as chaves cujo `JSON.stringify(antes[k]) !== JSON.stringify(depois[k])`,
      com linha `-` (antes, vermelho) e `+` (depois, verde).
- **Loading states:** botoes de Salvar/Excluir ficam disabled durante request
  com label trocado (`Salvando...`, `Excluindo...`). Estado `submitting` compartilhado.

### Componentes auxiliares (mesmo arquivo)
- `MotivoField`: textarea reutilizavel (2 rows).
- `HistoryEntryCard`: card colapsavel.
- `DiffView`: roteia entre 3 modos (`create`/`update`/`delete`).
- `PayloadKv`: render simples key-value de payload.
- `formatValue`: helper pra null/undefined/strings/objetos.

## Comportamento Pos-Fix
- **Auditor mestre nao consegue salvar sem motivo.** UI rejeita; backend ja
  aceitava string vazia, agora a barreira de qualidade fica na borda do front.
- **Toda mutacao deixa rastreabilidade visivel**: clicar no icone de historico
  mostra quem mudou o que e quando, com diff visual antes/depois.
- **POP ref editavel pela UI**: auditor consegue reapontar um alerta pra outro
  POP sem PR (e o `_audit_log` registra a mudanca).
- **Delete agora sempre tem motivo**: nao da mais pra apagar setor "por
  engano" sem deixar trail.

## Riscos
- **Sem teste automatizado pra UI**: o projeto nao tem suite de testes de
  componente React. Validacao manual obrigatoria pos-deploy (ver checklist).
- **`payload_antes/depois` pode ter chaves novas conforme o backend evolui**:
  o `DiffView` lida com qualquer estrutura JSON (diff por `JSON.stringify`),
  mas se o backend mudar nomes de coluna sem migrar audit_log antigo, o diff
  pode ficar confuso. Aceita-se ja que payload_antes e snapshot historico.
- **lru_cache multi-pod (carregado da Fase 1.1)**: editar via UI invalida so
  o pod que recebeu a request. Em Cloud Run com >1 pod, IA dos outros pods
  pode usar catalogo antigo ate proxima rotacao. Conhecido, documentado em
  `1.3.68`.
