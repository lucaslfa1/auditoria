# v1.3.213 — Transcrição: correção fuzzy fonética + prompt Whisper dentro da janela de 224 tokens

**Data:** 2026-07-02
**Tipo:** Fix (transcrição/anti-alucinação) — "Opentech" virando "pintech" e afins
**Branch:** `main`

## Contexto

Mesmo com o dicionário heurístico (60+ padrões regex só para "Opentech" em
`config/text_corrections.json`), a transcrição seguia produzindo variantes novas
que passavam direto — caso relatado em 02/07: **"pintech"**. O dicionário regex é
"whack-a-mole": cada erro inédito do ASR exige um padrão novo, e o espaço de
variantes fonéticas é infinito.

## Causa raiz (a mais grave): prompt do Whisper truncado silenciosamente

O Whisper (Azure OpenAI) considera **apenas os ÚLTIMOS 224 tokens** do parâmetro
`prompt` (documentado na transparency note da Azure OpenAI). Os prompts de
vocabulário em `ai_prompts` (`whisper_prompt.default` e `whisper_prompt.<setor>`,
expandidos na m20260527_001) têm **700+ tokens** e começam justamente com
"Opentech, nstech, BAS, Mondelez..." — ou seja, **o glossário crítico estava
sendo descartado** pelo modelo e só a cauda do prompt (rodovias, facções, URA)
chegava ao Whisper. O viés de vocabulário que deveria evitar "pintech" nunca
esteve ativo no caminho Whisper.

## Mudança

1. **`fit_prompt_to_whisper_window`** (`transcription_providers/common.py`):
   reordena o prompt colocando os termos críticos (novo
   `whisper_prompt_priority_terms` do text_corrections.json; fallback = targets
   das correções) **no FIM** — a parte que sobrevive à janela de 224 tokens — e
   corta o excedente do início (estimativa conservadora ~3 chars/token, budget
   210). Aplicado em `_resolve_whisper_prompt` (core/transcription.py), cobrindo
   prompts do banco E o fallback local. Termos prioritários ausentes no prompt
   são adicionados; duplicatas removidas sem distinção de caixa.

2. **Camada fuzzy fonética** (`utils/text_processing.py`), nova seção
   `fuzzy_corrections` no text_corrections.json: depois dos regex,
   `normalize_company_name` compara a forma fonética pt-BR de cada token (e de
   pares adjacentes, ex.: "pin tech") com os alvos canônicos (Opentech, Mondelez,
   Omnilink, Autotrac, Onixsat, Maxtrack, Unilever, Buonny, nstech) por distância
   de edição com tolerância proporcional ao tamanho. Generaliza o dicionário para
   variantes **nunca vistas** ("pintech", "opantech", "openteki"...) sem custo de
   API. Guard-rails: `protected_words` (gente, modelo(s), sigla, sacar, fintech,
   pente...), tamanho mínimo de token (5) e distâncias conservadoras — Sighra e
   Sascar ficaram FORA dos alvos fuzzy de propósito (colidiriam com "sigla" e
   "sacar"; os regex existentes já os cobrem).

3. **Regex novo** para a família "pintech": `\b[Pp]i[nm]\s*te(?:ch|ck|c|k|x|que)\b`
   (pintech, Pimtech, pin tech, pinteck...).

4. **Custom Speech opcional no Fast Transcription** (`transcription_providers/azure.py`):
   se `AZURE_SPEECH_CUSTOM_MODEL_URI` estiver definido, a definition passa
   `"models": {"pt-BR": <uri>}` (suportado na api-version 2025-10-15). É o caminho
   definitivo para o engine primário (`fast`): modelo treinado com corpus do
   domínio + pronúncias. Sem a env, comportamento idêntico ao atual.
