# v1.3.90 — Save audit responde em ~50ms (RAG em background) + senha ditada digito por digito

**Data:** 2026-05-27
**Tipo:** Performance + correção de avaliação IA
**Custo:** zero

## Contexto

Lucas (auditoria):
1. "Quando editamos alguma auditoria ele demora pra salvar e também parece que da uma travada"
2. "peguei uma situação onde o motorista passou a senha corretamente mas foi zerada porque a transcrição separou os numeros por caracteres 02, 5, 5 e a IA provavelmente entendeu errado em vez de considerar 0255 (4 numeros, senha padrao) ela zerou"

### Problema 1: save lento (BUG-???)

`update_audit_by_id` chamava `salvar_feedback_rag_sync` **DENTRO** da request HTTP, ANTES do commit. Quando o auditor mudava status de critério (auditoria errou e foi corrigida), o pipeline RLHF gerava embedding via Azure OpenAI (~200ms a vários segundos sob throttling) e SÓ DEPOIS o response saía.

Pior: a transação SQL ficava aberta durante a chamada externa, segurando conexão do pool.

### Problema 2: senha 4 dígitos zerada por dígitos ditados separados

Motorista falou "0 2 5 5" ou "02, 5, 5". STT separou os dígitos em tokens. IA leu "02" + "5" + "5" e ou:
- Achou que eram 3 grupos (menos de 4 dígitos) → senha errada
- Achou que era "02" como senha de 2 dígitos → errado
- Aplicou flag `senha_incorreta_aceita` → zerou a auditoria

A senha real era 0255 (correta, 4 dígitos), mas a IA não juntou os fragments.

## Decisões

### Problema 1: BackgroundTasks do FastAPI
- `update_audit_by_id` no repository agora **NÃO** chama RAG. Retorna `dict {"updated", "rag_payload"}` onde `rag_payload` carrega o dado pra processar depois.
- `routers/saved_files.atualizar_salvo` recebe `BackgroundTasks` e agenda `_run_rag_feedback_background(payload)` DEPOIS do `return`.
- Resultado: PUT responde em ~50ms (só UPDATE SQL); embedding + insert RAG rodam após o response no mesmo container.

Por que BackgroundTasks e não thread daemon: no Cloud Run, threads daemon morrem quando o request termina. BackgroundTasks do FastAPI mantém o container vivo até a task acabar — funciona corretamente.

### Problema 2: prompt explícito para juntar dígitos ditados
- Em `backend/config/prompts.json::audit_system.regra_senha`, adicionada sub-regra "REGRA DE DIGITOS DITADOS" com exemplos:
  - `'02, 5, 5'` → `0255`
  - `'0, 2, 5, 5'` → `0255`
  - `'zero dois cinco cinco'` → `0255`
  - `'02 55'` → `0255`
- Instrução explícita: contar dígitos APÓS concatenar. Se ficar com 4 dígitos = tamanho correto. Em dúvida razoável (menos), preferir PASS.

Decisão de fazer no prompt e não no preprocessing: o LLM já lida bem com contexto ambíguo quando instruído. Preprocessing regex separaria casos legítimos (ex: motorista dita CPF de 11 dígitos e operador confunde com senha) — confiar no LLM ler o contexto é mais robusto.

## Comportamento esperado

1. **Salvar edição de auditoria**: response em ~50ms (vs 500ms-5s antes). Auditor não sente travada. Feedback RAG continua sendo gerado, só não bloqueia mais.
2. **Motorista dita senha "0 2 5 5"**: IA concatena pra `0255`, conta 4 dígitos, status PASS no critério de senha — NÃO zera.
3. Se motorista realmente passar senha errada (5+ dígitos concatenados, ou senha curta sem fallback), IA continua marcando senha_incorreta_aceita corretamente.
