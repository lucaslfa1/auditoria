# v1.3.106 - Setores editáveis (rename + cascata para colaboradores)

**Data:** 2026-06-03
**Tipo:** Feature - gestão de setores + Bugfix (item 4: acento em `sector_id`)
**Status:** implementado e testado (puro); integração aguarda banco de teste; aguardando deploy (CI/CD) + validação

## Contexto

Pedido do Lucas: tornar **setor** customizável como operadores/supervisores — uma
tela onde o setor pode ser criado/renomeado e, ao renomear, **todos os colaboradores
vinculados também mudam**, mas a **regra de auditoria se mantém**.

Diagnóstico (a infra já existia ~70%):
- `audit_sectors` (`id` TEXT PK fixo + `label` + `description`) é a âncora das regras:
  `audit_alerts.sector_id` → FK para `audit_sectors.id`. As regras se prendem ao **id**,
  nunca ao rótulo.
- `colaboradores.setor` é string livre ("FÊNIX", "UTI"). O "vínculo" é implícito:
  `setor` → mapa de apelidos (`sector_aliases`) → `audit_sectors.id`. **Nunca pelo rótulo.**
- Buracos: (1) renomear o setor mexia só em `audit_sectors.label`, **não** cascateava
  para `colaboradores.setor`; (2) a gestão estava espalhada entre admin Critérios e
  Apelidos, sem uma tela limpa.

Por o vínculo passar pelo `id` + apelidos, **renomear o rótulo é incapaz de quebrar
um critério** — o requisito "regra se mantém" sai de graça.

## Mudança

**Backend — `repositories/sector_aliases.py`:** extraído `match_canonical_sector(rules, ...)`
(matcher puro, sem I/O) de `resolve_canonical_sector` (que agora delega — comportamento
idêntico) + `list_active_rules()` público. Permite resolução em lote dentro de uma
transação sem reabrir conexão por item.

**Backend — `repositories/admin_criteria.py`:**
- `get_sector_members(factory, sector_id)` — read-only, fonte única da associação
  funcionário→setor (reverse-resolution via apelidos). Usada pelo preview e pela cascata.
- `rename_sector_with_cascade(factory, sector_id, new_label, ..., cascade)` — numa única
  transação: `UPDATE audit_sectors.label/description` + (se `cascade`) bulk
  `UPDATE colaboradores.setor` dos membros com snapshot em `colaboradores_audit_log` +
  garante alias `setor_exact` (`_norm(new_label)` → `sector_id`, prioridade 200, p/ o
  novo nome seguir resolvendo ao mesmo id) + log em `audit_sectors_audit_log`. O `id`
  NUNCA muda.

**Backend — `routers/admin_criteria.py`:** `GET /api/admin/sectors/{id}/members`
(preview "N afetados") + `POST /api/admin/sectors/{id}/rename` (`new_label`,
`description?`, `cascade`). `require_admin`. Model `SectorRename`.

**Backend — `db/database.py` (item 4):** em `persist_audit_artifacts`, antes do
`save_audit`, canonicaliza `effective_sector_id` via `resolve_canonical_sector`
(fail-soft: mantém a forma crua se nada casar). Corrige acento/forma inconsistente em
`audits.sector_id` **apenas nas escritas novas**; não reescreve histórico.

**Frontend — `src/features/admin/components/SectorManagement.tsx` (novo):** tela de
Setores (espelha `AdminSectorAliasesPage`/`OperadorManagement` — dark mode, tabela +
modais + toast). Lista setores com contagem de vinculados; criar; renomear com toggle
"aplicar aos colaboradores vinculados" + preview de quantos serão afetados. Wired em
`App.tsx` (view `admin-sectors`, admin-gated) + item "Setores" no submenu IA da Sidebar.

## Decisão de escopo (Tier 1)

`id` interno **fixo e invisível** (chave das regras); usuário edita rótulo + associação.
Excluído desta entrega: reescrever `audits.sector_id` histórico; criar setor com critérios
próprios pela UI; merge de setores.

## Riscos / BI

- Renomear rótulo (`audit_sectors.label`): risco ZERO — Fechamento/BI não leem `audit_sectors`.
- Cascata (`colaboradores.setor`): risco BAIXO — `_normalize_setor_fechamento` decide
  quase tudo por `escala`; `setor` é só fallback/token. Verificar agrupamento de
  Fechamento antes/depois ao renomear um setor reconhecido por token no `setor`.
