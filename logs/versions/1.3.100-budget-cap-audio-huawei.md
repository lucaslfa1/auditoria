### Conclusoes do benchmark

1. **`hybrid_dual` instavel** — 3 falhas em 4 runs (75%). Sempre por "fusao
   GPT-4o em modo estrito (merge_rejected_diagnostics)". Nao usar em prod.
2. **`fast`/`whisper`/`gpt4o_diarize` convergem** — fallback chain do `fast`
   cai pro `whisper` quando precisa, produzindo texto identico em 2/3 dos
   audios. `gpt4o_diarize` se diferencia (12% mais texto) apenas em audios
   longos.
3. **Tempo escala com duracao do audio**, nao com engine:
   - 3 min audio -> ~190-220s
   - 5 min audio -> ~200s
   - 12 min audio -> ~470-520s
4. **O gargalo nao e a engine.** Mudar `AZURE_TRANSCRIPTION_ENGINE` no Cloud
   Run nao destrava nada (em prod ja resolve pra `fast` por causa de
   `_LEGACY_HYBRID_DUAL_ALIASES`). O problema e o cap `time_budget=540s` que
   nao acomoda um unico audio Huawei longo.

A linha 593 de `automation.py`:
```python
timeout_seconds=min(item_timeout_seconds, max(1.0, deadline - time.monotonic()))
```
Passa pro item o **tempo restante do budget**. Com budget 540s e item medio
~480s, o 2o item recebe ~60s e morre. Erro engana ("Timeout apos 60s")
sugerindo bug de env, mas eh design — deadline acabando.

## Mudancas

### `backend/core/automation.py`

```python
# _get_automation_item_timeout_seconds: cap 540 -> 900
return max(60, min(parsed, 900))

# _get_automation_audit_time_budget_seconds: cap 540 -> 1800
return max(60, min(parsed, 1800))
```

Cap 1800s = 30 min: cabe 3-4 audios Huawei tipicos (~500s cada) no pior
caso. Cap 900s no item_timeout cobre o pior caso observado (520s) com
margem.

### `tests/backend/test_automation_item_timeout.py`
- `test_cap_max_540` virou `test_cap_max_900`.

### `tests/backend/test_automation_time_budget.py` (novo)
- 4 testes para `_get_automation_audit_time_budget_seconds`: env wins,
  fallback DB, cap max 1800, cap min 60.

### Configs no Neon (`auditoria-nstech-2`)

```sql
UPDATE configuracoes SET valor = '1500'
 WHERE chave = 'automacao_audit_time_budget_seconds';
UPDATE configuracoes SET valor = '600'
 WHERE chave = 'automacao_item_timeout_seconds';
```

Valores dentro do cap (com margem). budget=1500s da 5 min de folga pra cron
de 30 min nao colidir; item_timeout=600s eh slightly above o pior caso
medido (~520s) sem ser tao alto que esconda bugs.

### Reabertura de 5 itens com `automation_timeout`

```sql
UPDATE fila_revisao_classificacao
SET status = 'auto_resolved',
    motivos_json = (SELECT jsonb_agg(value)::text
                    FROM jsonb_array_elements_text(motivos_json::jsonb)
                    WHERE value <> 'automation_timeout')
WHERE id IN (68, 73, 78, 79, 83)
  AND status = 'pending'
  AND alerta_previsto NOT IN ('desconhecido', 'erro');
```

5 itens (Raffaella 68, Thais 73, Loren 78, Asdrubal 79, mais 83) voltaram
pra `auto_resolved` pra entrar no proximo ciclo automatico apos deploy.
