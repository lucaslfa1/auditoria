# v1.3.121 — Scripts e runbook de migração do banco (Neon → Azure PostgreSQL)

**Data:** 2026-06-11
**Tipo:** Handover — pacote de migração de banco
**Custo:** nenhum (artefatos de migração; nada roda em produção)
**Branch:** `handover`

## Entregáveis

### `scripts/migration/` (novo)
- `01_dump_neon.sh` / `.ps1` — `pg_dump -Fc --no-owner --no-privileges`;
  força endpoint direto (remove `-pooler`); exige cliente >= 17.
- `02_restore_azure.sh` / `.ps1` — `pg_restore -j 4 --no-owner` com os
  pré-requisitos documentados (pgvector via `azure.extensions` ANTES).
- `03_validate.py` — compara origem×destino: tabelas, contagem de linhas,
  `schema_migrations`, sequences seriais críticas e presença do pgvector.
  Exit 1 em qualquer divergência.

### `docs/10-migracao-azure-db.md` (novo)
Runbook completo: fatos da origem (PG 17.10, 46 MB, 44 tabelas, extensões
`vector 0.8.0` obrigatória + `pg_stat_statements` opcional), pré-requisitos
Azure, janela passo a passo (congelar cron/automação → dump → restore →
validar → apontar `DATABASE_URL` → smoke → descongelar; ~15 min p/ 46 MB),
rollback (Neon intocado), particularidades (pooler, sslmode, sequences,
alternativa banco-novo-do-zero via seed v1.3.120).

## Ensaio executado (PG16 local)
Dump do banco de teste (38 tabelas, 53 migrations) → restore em banco vazio →
`03_validate.py`: checks 1-4 ✅; check 5 (pgvector) ❌ **de propósito** — o
PG local não tem a extensão, provando que a validação pega exatamente a
configuração que faltaria no Azure. Artefatos do ensaio removidos.

## Correção do plano original
O plano dizia "zero extensões PostgreSQL" — **errado**: `vector` (pgvector
0.8.0) é exigida pelas tabelas de RAG. Runbook e validação cobrem isso.
