## Contexto

`v1.3.89` introduziu pause/resume e cancel hard com estado em memoria
(`_LAST_SYNC*` globals em `routers/telefonia.py`). Documentou na propria
release:

> Estado vive em memoria do processo (Cloud Run). Se o pod reiniciar durante
> pausa, o sync se perde â€” auditor precisaria disparar novo manual.

`v1.3.91` reforcou a pendencia: ao reiniciar, a UI mostrava "Coleta rodando"
indefinidamente porque o status persistia apenas em memoria do processo
anterior.

Lucas escolheu (2026-05-28) a opcao minima: persistir **status + heartbeat
+ flags de controle**, sem checkpoint de cursor de retomada. Reinicio nao
tenta continuar de onde parou â€” auditor redispara â€” mas a UI volta a
mostrar estado coerente.

## Decisoes

1. **Onde guardar**: extender `telefonia_sync_history` (sem nova tabela).
   - `pause_requested BOOLEAN DEFAULT false`
   - `cancel_requested BOOLEAN DEFAULT false`
   - `last_heartbeat_at TIMESTAMPTZ`
2. **Granularidade**: status + heartbeat. Sem persistir lista de candidatos
   processados nem cursor de retomada.
3. **Reconcile**: bootstrap do FastAPI marca runs com `finished_at IS NULL`
   e `last_heartbeat_at < now - 120s` como `status='interrupted'`,
   `finished_at=CURRENT_TIMESTAMP`.

### `backend/routers/telefonia.py`
- Novo global `_LAST_SYNC_RUN_ID` armazena o id do run no DB.
- `sync_manual` chama `start_telefonia_sync_run` antes de criar a `asyncio.Task`.
- `_run_manual_sync` faz heartbeat dentro do `progress_callback`
  (qualquer mudanca de estagio/progress atualiza `last_heartbeat_at`).
- Wrapper `_finalize()` substitui os 3 `save_telefonia_sync_history` antigos
  por `finalize_telefonia_sync_run(run_id)`. Fallback: se `run_id` nao foi
  obtido (DB indisponivel no start), grava via API legada.
- `pause_sync` chama `set_telefonia_sync_pause(true)` + heartbeat com `status='paused'`.
- `resume_sync` chama `set_telefonia_sync_pause(false)` + heartbeat com `status='running'`.
- `cancel_sync` chama `set_telefonia_sync_cancel(true)` + `set_telefonia_sync_pause(false)`
  + heartbeat com `status='cancelling'`.

### `backend/main.py`

## Comportamento esperado

1. **Sync rodando + pod morre**: bootstrap marca o run como 'interrupted'.
   A UI puxa `/sync/status` e mostra o resultado coerente; auditor decide
   redisparar.
2. **Sync pausado + pod morre**: idem â€” run vira 'interrupted'. Auditor
   redispara (resume automatico nao existe nesta versao).
3. **Sync cancelado**: `cancel_requested=true` no DB antes da task ser
   abortada; a UI sabe diferenciar 'cancelling' de 'failed' mesmo apos
   restart.
4. **Heartbeat normal**: `progress_callback` atualiza `last_heartbeat_at`
   a cada mudanca de estagio (`discovered`, `downloading`, `enfileiramento`,
   etc.). Tipicamente a cada poucos segundos durante o sync ativo.
