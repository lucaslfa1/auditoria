**Branch:** `handover`

## Mudanças

### 1. Env `AUDIT_WEAK_EVIDENCE_RETRY` (`backend/core/audit_evaluator.py`)
O retry da avaliação por evidência fraca (cobertura < `AUDIT_MIN_MATCHED_EVIDENCE_RATIO`,
default 0.72) dobra o custo de GPT-4o do item. Agora é configurável:
- **default `1`** — comportamento histórico preservado (1 retry);
- **`0`** — payload VÁLIDO com evidência fraca é aceito com warning, sem a
  segunda chamada. Payload INVÁLIDO continua tendo retry (sem ele não há
  resultado).
O retry, quando ocorre, já é contado no `cost_guard` (categoria
`avaliacao_retry`, desde v1.3.114) — dá para medir no `custo_diario` quanto
essa alavanca custaria antes de decidir desligar.

### 2. Polling pausa com aba oculta (`src/features/classifier/components/RemoteTriageQueue.tsx`)
O interval de 5s do `/audit-status` agora não dispara com `document.hidden`
— era o único polling do front sem essa guarda (custo de compute Neon).

### 3. Decisão documentada: `AZURE_GPT4O_DIARIZE_SMART_ROUTING` fica como está (default ON)
O plano original previa desligar por default, assumindo que o prescan de 35s
era custo extra. Validação no código mostrou o INVERSO
(`transcription.py:_should_use_gpt4o_diarize_as_primary_for_audio`):
- o caminho só ativa para setores/alertas em `AZURE_GPT4O_DIARIZE_PRIMARY_SECTORS`
  (hoje VAZIO em prod → caminho dormente, custo zero);
- quando ativo, o prescan (1 chamada Fast barata em 35s) decide se o
  gpt4o-diarize caro é necessário; smart routing OFF = gpt4o-diarize SEMPRE
  primário = MAIS caro.
Conclusão: desligar pioraria o custo. Default mantido; comportamento
documentado para o time de engenharia em docs/07-custos-e-guardrails.md
(etapa de documentação).
