## Objetivo
Neutralizar o seed destrutivo do catalogo de criterios. Apos a Fase 1.1
(`1.3.68`), o DB era a fonte primaria mas o seed ainda UPSERTava o conteudo do
YAML em todo deploy — sobrescrevendo qualquer edicao via UI. Esta versao quebra
essa rede de seguranca e torna as edicoes admin permanentes.

## Contexto
A Fase 1.1 (1.3.68) deixou o `_seed_audit_criteria` ainda rodando em todo deploy
como rede de seguranca. **Sem este passo, todo deploy futuro reverte qualquer
mudanca feita pelo auditor mestre na UI.** Esta versao fecha esse loop.

## Mudancas

### `backend/database.py:_seed_audit_criteria`
- **Aborto curto:** primeira coisa que faz e `SELECT COUNT(*) FROM audit_sectors`.
  Se for > 0, retorna imediatamente com log INFO "Catalogo ja populado, seed
  ignorado (preservando DB)".
- **Sem mais DELETE/UPSERT destrutivo:** os `DELETE FROM audit_criteria WHERE
  alert_id NOT IN ...` e `ON CONFLICT(id) DO UPDATE SET ...` foram substituidos
  por `INSERT ... ON CONFLICT(id) DO NOTHING`.
- **Hash check removido:** nao precisa mais comparar hash do YAML, ja que so
  roda em DB vazio.
- **Resultado:** o seed agora so se ativa em bootstrap (DB recem-criado em local
  dev ou em uma branch nova do Neon). Em producao, apos o primeiro seed
  realizado, fica permanentemente inerte.

### `backend/db/scoring_rules.yaml` → `backend/db/seeds/scoring_rules.bootstrap.yaml`
- Arquivo movido + renomeado para deixar claro arquiteturalmente: **so vale
  para bootstrap inicial**.
- `backend/db/scoring_loader.py:_YAML_PATH` atualizado:
  ```python
  _YAML_PATH = Path(__file__).resolve().parent / "seeds" / "scoring_rules.bootstrap.yaml"
  ```

### `backend/database.py` – Conteudo do seed
- `_alerts` agora monta `alert_rows` com 5 campos (`id, sector_id, label, context,
  pop_ref`), bem como o INSERT correspondente. Mantida compatibilidade com a tupla
  expandida de `scoring_loader.build_seed_data_from_yaml`.

## Comportamento Pos-Fix
- **Producao:** `_seed_audit_criteria` ja foi executado uma vez (apos a Fase 0/
  1.1). Em todo deploy futuro, o log diz "Catalogo ja populado. Seed ignorado."
  e termina em ms. Edicoes admin via UI sao permanentes.
- **DB novo (dev/branch nova):** o seed carrega o YAML de bootstrap uma vez.
  Apos isso, mesmo nesse ambiente, fica inerte.
- **YAML como referencia:** o `seeds/scoring_rules.bootstrap.yaml` ainda existe
  e e o snapshot autoritativo do POP no momento desta versao — mas perde o
  status de "fonte da verdade em runtime".
- **Auditor mestre pode editar critericos via UI** e as mudancas persistem entre
  deploys. Trail completo em `audit_*_audit_log` (introduzido em 1.3.68).

## Riscos
- **Sem rede de seguranca:** se alguem deletar tudo de `audit_sectors`/
  `audit_alerts`/`audit_criteria` por engano em producao, o seed nao restaura
  automaticamente — sera necessario rodar manualmente o seed (ou aplicar backup
  do Neon). Esse era o comportamento implicitamente prometido pelo seed antigo,
  porem destrutivo. Mitigacao: Neon tem snapshots automaticos + audit_log
  permite reconstruir.
- **Divergencia YAML vs DB:** se alguem editar so o YAML, o DB nao reflete e
  a IA nao usa. Vale documentar internamente que o YAML virou snapshot
  historico/bootstrap e nao deve ser editado para mudanca operacional.
