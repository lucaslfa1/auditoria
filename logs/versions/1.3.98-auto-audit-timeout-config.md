# v1.3.98 — Auditoria automatica timing out: aumenta budget + batch e pede env

**Data:** 2026-05-28
**Tipo:** Configuracao + diagnose (zero linha de codigo Python)
**Custo:** deploy backend NAO necessario; CI/CD precisa setar 1 env var no Cloud Run

## Contexto

Lucas (auditoria) testando automacao desde 11h hoje: items sao baixados pelo
Cloud Scheduler (cron a cada 30min), entram em Triagem como
`auto_resolved`/`pending` com `class_status='done'`, mas **NENHUM aparece em
Arquivos Salvos**. Os ciclos terminam com status 'partial' e 0 auditadas:

| Ciclo | Hora    | Items tentados | Auditadas | Erro                                                              |
|-------|---------|----------------|-----------|-------------------------------------------------------------------|
| 780   | 14:15Z  | 2              | 0         | Timeout em Raffaella (~480s)                                      |
| 781   | 14:45Z  | 3              | 0         | Timeout em Caio (~480s)                                           |
| 782   | 15:15Z  | 3              | 0         | Timeout em Loren (~155s) + 1 blocked                              |
| 783   | 15:45Z  | 2              | 0         | Timeout em Thais (~480s)                                          |

Lucas: "precisamos deixar essa auditoria automatica funcionando 100%, eh o
foco principal do projeto — fluxo vai sozinho ate Arquivos e Auditorias do mes."

## Diagnostico

Mismatch de timeouts no pipeline:

| Variavel                                | Default | Onde                                 |
|-----------------------------------------|---------|--------------------------------------|
| `AUTOMATION_ITEM_TIMEOUT_SECONDS`       | **480s**| automation.py:289 (per-item budget)  |
| `AZURE_TRANSCRIPTION_TIMEOUT_SECONDS`   | **600s**| transcription via config.py:283      |
| `AZURE_TRANSCRIPTION_STRATEGY_TIMEOUT_SECONDS` | **840s** | transcription.py:497 (com fallbacks) |
| Cap hard-coded do item timeout          | **540s**| automation.py:294 `min(parsed, 540)` |

480 < 600 < 840 → o item budget (480s) eh **menor** que o timeout interno da
chamada Azure. Por design, qualquer chamada Azure que demore +480s mata o item
antes do Azure responder.

Audit_result do ciclo 783 confirma:
- `last_step_at` apenas 5s depois do `started_at` (ultimo heartbeat foi
  `transcribing_and_evaluating_audio`)
- `finished_at` ~480s depois -> `time_budget_exhausted=true`
- Nenhum heartbeat por 8 minutos = pipeline ficou preso dentro de
  `process_audit_with_ai` (transcricao+avaliacao IA)

Items que foram auditados com sucesso (id 40, 55, 56) vieram via botao
"Auditar instantaneamente" (manual), nao via automation cycle.

### Reabertura dos itens com `automation_timeout`

4 itens estavam em `status='pending'` apos timeout (Loren #78, Caio #74,
Thais #73, Raffaella #68) com alerta valido. UPDATE devolveu pra
`auto_resolved` (que entra no filtro virtual `ready_for_audit`) e removeu
`automation_timeout` de `motivos_revisao`. Proximo ciclo do Cloud Scheduler
deve pegar os 5 (67 + 68 + 73 + 74 + 78).

```sql
UPDATE fila_revisao_classificacao
SET status = 'auto_resolved',
    motivos_json = (SELECT jsonb_agg(value)
                    FROM jsonb_array_elements_text(motivos_json::jsonb)
                    WHERE value <> 'automation_timeout')::text,
    atualizado_em = NOW()
WHERE id IN (78, 74, 73, 68)
  AND status = 'pending'
  AND alerta_previsto NOT IN ('desconhecido', 'erro');
```

### Env var pendente (responsabilidade do deploy automático)

**Cloud Run env var a setar:**
```
AZURE_TRANSCRIPTION_TIMEOUT_SECONDS=300
```

Justificativa: precisa ser **menor** que `automacao_audit_time_budget_seconds`
(540s) com margem. 300s da margem de 240s para steps preparatorios
(metadata, operator gate, quota, audio load, validation, cache, hash) que
hoje rodam em ~5s mas podem variar.

Default da config eh 600s — sem essa env, mesmo com budget 540s o pipeline
continua sendo morto antes do Azure responder.
