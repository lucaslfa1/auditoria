# v1.3.120 — Seed do catálogo oficial de critérios (banco novo completo)

**Data:** 2026-06-11
**Tipo:** Infraestrutura de banco / handover
**Custo:** nenhum em prod (guarda só ativa em banco com <= 1 setor)
**Branch:** `handover`

## Contexto

Banco recém-criado ficava só com o setor `logistica` do seed legado
(`_seed_audit_criteria`). O catálogo real de produção (12 setores, 71
alertas, 1051 critérios — ground truth mantida pela auditora oficial) foi
construído via UI/imports e não tinha seed. Consequências:
1. O time de engenharia não conseguiria subir um ambiente funcional do zero.
2. 15 testes de guardrail de classificação falhavam em banco de teste limpo
   (metade das 28 falhas do baseline do handover).

## Mudanças

### `backend/db/seeds/audit_catalog_oficial.sql` (novo, 1134 INSERTs)
Dump `--data-only --column-inserts --on-conflict-do-nothing` das tabelas
`audit_sectors`, `audit_alerts`, `audit_criteria` do banco de produção
(2026-06-11). `--column-inserts` garante independência da ordem física de
colunas (prod evoluiu via ALTERs).

### `backend/db/database.py` — novo `_seed_official_catalog(c)`
Chamado no `init_db()` após o seed legado:
- **Guarda**: só roda com `count(audit_sectors) <= 1` (banco novo). Banco
  real (>= 2 setores) é no-op — edições via UI intocadas.
- Em banco novo: remove o seed legado antes (evita colisão de ids seriais
  que o ON CONFLICT esconderia) e aplica o dump.
- Filtra meta-comandos psql (`\restrict`/`\unrestrict` do pg_dump 18) e o
  cabeçalho de SETs (`transaction_timeout` é PG17+; quebraria em PG16).
- Realinha a sequence serial de `audit_criteria` pós-INSERTs com id explícito.

(comando no cabeçalho do arquivo .sql) — ou o time pode promover o catálogo
para migração formal depois.
