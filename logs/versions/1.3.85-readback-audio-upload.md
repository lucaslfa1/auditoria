# v1.3.85 — Read-back após upload de áudio (invariante "audit só com download confirmado")

**Data:** 2026-05-27
**Tipo:** Defesa em profundidade / consistência GCS↔DB
**Custo:** zero (caminho feliz inalterado)
**Bug relacionado:** BUG-034 (auditorias com `audio_storage_path` no DB apontando para blob ausente no GCS)

## Contexto

Usuário (Lucas) reportou: "muitas ligações são auditadas sem termos os arquivos". Investigação:

- DB Neon mostra 10 audits com `source_type='audio'`, todas com `audio_storage_path` e `audio_size_bytes` preenchidos.
- GCS (`auditoria-nstech-audios`) tem **8 desses 10 blobs** com size batendo.
- **audit #3** (`2026/05/audit_3_bb536d3a69bd.wav`, 1.5MB declarado) — **NÃO existe no GCS** nem em soft-delete.
- **audit #4** (`2026/05/audit_4_77ebc22a2276.mp3`, 443KB declarado, único MP3) — **NÃO existe no GCS** nem em soft-delete.

Ambos criados em 26/05 entre 16:23 e 16:29 — janela curta sugerindo problema transitório. Pipeline atual tem guards (`ClassifiedAudioUnavailableError` em `core/automation.py:825`, `if not audio_bytes: return None` em `db/database.py:791`), mas houve buraco: `save_audit` insere row ANTES de `store_audit_audio_file` confirmar upload. Se o upload silenciar uma falha (ex: Cloud Run restart entre as operações, exception capturada implicitamente), o audit fica órfão.

## Estratégia (conservadora, conforme escolha do Lucas)

Adicionar **read-back pós-upload** sem reordenar `save_audit` → `store_audit_audio_file`. Isso garante que `store_audit_audio_file` só retorna sucesso se conseguir confirmar que o blob existe com o tamanho esperado. Caller continua igual; se `AudioUploadVerificationError` for levantada, `update_audit_audio_storage` não é chamado e a row permanece com `audio_storage_path = NULL` (estado verificável, não "fantasma").

Reordenar `save_audit`/`store_audit_audio_file` para tornar atômico fica como pendência (decisão deliberada do Lucas para reduzir risco de regressão agora).

## Mudanças

### `backend/storage/audit_storage.py`
- Nova exception `AudioUploadVerificationError(RuntimeError)`.
- `store_audit_audio_file`:
  - **GCS path**: após `blob.upload_from_string`, faz `blob.reload()` + verifica `blob.size == len(audio_bytes)`. Se reload falha ou size diverge, deleta o blob e levanta `AudioUploadVerificationError`.
  - **Local path**: após `os.replace`, faz `absolute_path.stat().st_size == len(audio_bytes)`. Se stat falha ou size diverge, remove arquivo e levanta `AudioUploadVerificationError`.
- Logger explicito (`logger = logging.getLogger(__name__)`) substitui o ad-hoc anterior.

### `backend/tests/test_audit_storage_files.py`
- Importa `AudioUploadVerificationError`.
- Novo teste `test_store_audit_audio_file_local_readback_size_mismatch_cleans_up`: força `stat.st_size` divergente via patch, confirma que a exception é levantada E o arquivo final é removido.

- 🟡 **Recovery dos 2 órfãos atuais (audit_3, audit_4)**: requer re-buscar do Huawei OBS (se ainda tiver) ou marcar como `media_lost`. Pendência separada.
- 🟡 **Status `aguardando_audio` na fila Huawei**: não relevante ao bug atual (audit já foi feito, áudio sumiu depois). Pendência separada para casos onde Huawei demora a publicar o áudio.
