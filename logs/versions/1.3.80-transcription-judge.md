# v1.3.80 — Judge LLM real no empate do selector de transcrição

**Data:** 2026-05-27
**Tipo:** Feature (qualidade de transcrição)
**Custo:** ~1 chamada GPT-4o por auditoria com empate (não toda auditoria)
**Bug relacionado:** BUG-030 (judge nunca executava apesar do gate `needs_judge_tiebreaker`)
**Memória de plano:** `project_judge_tiebreak_2026_05_26.md`

## Contexto

Auditoria #10 (Jackeline Muniz da Silva, UTI, alerta `UTI-PRIORITARIO-MOT`) expôs que o selector de candidatos marcava `selection_status=needs_review` com `selection_reason=empate_requer_judge`, mas **o judge nunca rodava** — `judge_score` ficava `null` em todos os candidatos. O selector caía no fallback "maior `deterministic_score` ganha" e escolhia `gpt4o_diarize` (14151) sobre `fast` (14085) por diff de 0.47%.

Esse caso de empate é exatamente onde o `gpt4o_diarize` tende a alucinar, e foi o que aconteceu: "Bruno, já queria me dar o PINTEC aqui", "Tem um Huawei Mate 2X11, P20 e P23" (modelos de celular inventados), "Se quiser uma arma para mim" (em vez de "você desalarma pra mim"). O `fast` tinha o texto fiel.

Decisão de 2026-05-26: implementar judge LLM real (rejeitadas alternativas de tie-break determinístico). Quando o selector já marca `needs_judge_tiebreaker=true`, é porque os dois candidatos são equivalentes em métricas determinísticas — só LLM consegue distinguir alucinação de transcrição fiel.

## Mudanças

### 1. `backend/core/transcription_judge.py` (novo)
- `JudgeOutcome` dataclass frozen com `winner_label`, `winner_candidate_id`, `confidence`, `reason`, `scores` (por candidate_id), `hallucinations`, `raw_response`. Propriedade `resolved` indica se há vencedor binário (A ou B).
- `judge_tie_break(candidate_a, candidate_b, *, alert_id, alert_label, sector_id, operator_label, driver_label, client=None, deployment=None)`:
  - Cliente AzureOpenAI síncrono (lazy via `_build_default_client` se não passado). Permite injeção via `client=` para testes.
  - Carrega prompt da chave `transcription_judge` em `PROMPTS_CONFIG` (DB-first com fallback `prompts.json`).
  - Serializa segmentos com `_serialize_segments` (trunca em 8000 chars/candidato pra controlar tokens).
  - Chama `chat.completions.create` com `response_format={"type": "json_object"}`, `temperature=0`, `top_p=1`.
  - Parse robusto: `winner` ∈ {A, B, tie}; se A/B sem `winner_candidate_id` explícito, deriva do candidato correspondente.
  - **Falha silenciosa**: qualquer erro (config ausente, cliente indisponível, JSON malformado, exceção da API) → `None`. Caller mantém o fallback determinístico existente.

### 2. `backend/core/transcription.py`
- `build_selector_result` virou `async def` (chama judge via `asyncio.to_thread`):
  - Quando `decision.reason == "empate_requer_judge"` e há ≥2 candidatos usáveis, pega top + runner_up por `deterministic_score` e chama `judge_tie_break`.
  - Se `outcome.resolved` e `winner_candidate_id` está em `candidate_artifacts`, troca `selected` para o vencedor do judge.
  - Anota `metadata["judge"]` com payload completo do veredito + `judge_results[candidate_id]` por candidato.
  - Marca `metadata["selected_reason"] = "judge_resolved"` quando o judge prevalece.
- `_candidate_to_metadata` aceita `judge_results: Optional[dict[str, dict[str, Any]]]` e injeta `judge_score`/`judge_reason` no dict do candidato quando presente.
- Sites de chamada (`transcription.py:1357` e `:1378`) atualizados para `await build_selector_result()`.
- Import: `from core.transcription_judge import JudgeOutcome, judge_tie_break`.

### 3. `backend/config/prompts.json`
- Chave `transcription_judge` consumida pelo módulo novo. Prompt já estava no arquivo (commitado anteriormente como órfão).
  - `system`: critérios de avaliação ordenados (ausência de alucinação > atribuição correta de falante > coerência semântica > números críticos > integridade das frases).
  - `user`: template com placeholders para alerta, setor, falantes e os dois candidatos (provider, candidate_id, score, texto).

### 4. `backend/tests/test_transcription_judge.py` (novo)
- 7 testes do `judge_tie_break`:
  - Vencedor A com cenário Jackeline (fast vs gpt4o_diarize alucinado) — sanity da auditoria #10.
  - Vencedor B com derivação de `winner_candidate_id` a partir do label.
  - Empate (`tie`) marca `resolved=False`.
  - Resposta malformada → `None`.
  - Exceção da API → `None`.
  - Templates ausentes em `PROMPTS_CONFIG` → `None`.
  - Candidato vazio → `None`.
- 2 testes do `_candidate_to_metadata` (campos `judge_score`/`judge_reason` injetados/omitidos conforme `judge_results`).

## Persistência

Sem migração necessária. `backend/repositories/transcript_candidates.py:75-77` (`persist_for_audit`) já lê `judge_score` e `judge_reason` direto do row dict — colunas existem na tabela `transcript_candidates`.

## Resultado

- O judge GPT-4o roda **somente** quando `decision.reason == "empate_requer_judge"` (gate canônico). Não onera auditorias normais.
- Em falha do judge (network, JSON ruim, sem credencial), o pipeline cai no fallback determinístico atual — sem quebrar.
- Metadata `metadata["judge"]` permite auditar o veredito do juiz e correlacionar `judge_score` aos `deterministic_score` para futura calibração do `tie_ratio_threshold`.
