# v1.3.191 — Lista de Telefonia: consulta de 30s → 0,14s (materializa metadata jsonb)

**Data:** 2026-06-18
**Tipo:** Correção (performance)
**Branch:** `main`

## Contexto

A aba **Telefonia** (lista "Gravações recentes da Huawei") estava demoradíssima para
carregar. Investigação descartou cold start: `EXPLAIN (ANALYZE)` na consulta real,
em produção, mediu **30,2s a frio e 30,6s a quente** (mesma duração no 2º run → não
é cache). Tabelas são pequenas: `fila_revisao_classificacao` = 157 linhas (137
`huawei_sync`), `colaboradores` = 208.

- **Causa raiz:** em `listar_fila_revisao_classificacao`
  (`backend/repositories/classification_review_queries.py`), os dois `LEFT JOIN
  LATERAL` que resolvem o operador oficial referenciavam `f.metadata_json::jsonb`
  diretamente. O candidato Huawei `= ANY(ARRAY[...13 chaves...])` ficava no filtro
  do seq scan de `colaboradores`, fazendo o Postgres **re-parsear o JSONB de cada
  linha da fila a cada linha de colaborador**: ≈ 137 × 208 × 13 ≈ **370 mil parses
  de JSONB**. O lateral `official_by_huawei` sozinho gastava ~220ms × 137 loops.
- **Por que "já foi mais rápido":** o custo cresce com o nº de itens `huawei_sync`
  na fila (multiplicador 137×) — degradou conforme a fila acumulou.
- O `EXPLAIN` mostrava `Seq Scan on colaboradores` com `Buffers: shared hit=178170`.

## Mudança

### Backend

- [backend/repositories/classification_review_queries.py](file:///C:/Users/lucas.afonso/projetos/auditoria/backend/repositories/classification_review_queries.py):
  a query da listagem agora materializa o cast `metadata_json::jsonb` **uma vez por
  linha** numa CTE `f_base AS MATERIALIZED (… f.metadata_json::jsonb AS _mj)` aplicada
  ANTES dos LATERAL joins. Os candidatos Huawei e o match por nome passam a ler
  `f._mj ->> '...'` (jsonb já parseado) em vez de re-parsear o texto. `_huawei_operator_id_candidates_sql()`
  atualizado para emitir `f._mj`. `ORDER BY`/`LIMIT` movidos para a CTE (os LATERAL
  são LEFT JOIN — não alteram quais linhas nem a ordem; só evita rodar os laterais
  em linhas que seriam descartadas).

## Validação (read-only, contra o banco real)

- **Performance:** consulta real da função caiu de **30.256ms → 143,9ms** (~210×).
  `Buffers: shared hit` de **179.720 → 2.180**. Plano agora usa `CTE Scan on f_base`.
- **Equivalência:** comparação linha a linha (id + todos os campos `official_*` +
  `is_oficial`) entre a versão antiga e a nova: **100 linhas, mesmos ids, 0
  diferenças**.
- **Testes:** `tests/backend/test_review_queue_contract.py` verde (28 passes),
  incluindo o novo `test_listar_query_materializes_metadata_jsonb_once` (guarda de
  regressão de performance). Os contratos existentes (`status`/`params`/substrings)
  seguem passando. (3 testes `test_audit_all_pending_*` falham por falta de banco
  de teste local — ambiente, sem relação com esta mudança.)
