# v1.3.165 — Extrai resolução/matching de setor canônico de operators.py

**Data:** 2026-06-16
**Tipo:** Refatoração conservadora (legibilidade p/ handover)
**Custo:** nenhum
**Branch:** `main`

## O que foi feito

Extraído de `backend/repositories/operators.py` (1243 linhas) o cluster coeso de
**resolução e matching de setor canônico do colaborador** para o módulo irmão
`backend/repositories/operator_sector_mapping.py` (~106 linhas movidas).

Símbolos movidos (6):
- `_get_sector_aliases_repo`, `_get_default_connection_factory`,
  `_load_sector_aliases_dict` (helpers privados, uso exclusivo do cluster)
- `_map_organizacao_telefonia_to_sector`, `map_db_sector_to_classification_sector`,
  `_matches_operador_sector`

São funções que delegam a `repositories.sector_aliases` (regras de setor em DB,
editáveis via UI) — um subdomínio distinto do CRUD de colaboradores. Dependência
única é `_normalize_lookup_text` (importado direto de `operator_normalization`,
sem ciclo). Imports de `sector_aliases`/`db.database` seguem em runtime dentro das
funções.

## Compatibilidade

Todos os 6 nomes são reexportados de `repositories.operators`
(`from repositories.operator_sector_mapping import (...)  # noqa: E402,F401`).
Callers internos (`resolve_auditable_colaborador`, `get_colaboradores_lookup`,
`upsert_colaborador_telefonia`) e a fachada continuam usando `operators.<nome>`.

## Prova (padrão da refatoração conservadora)

- `ast.dump` byte-a-byte das 6 funções: idêntico entre `HEAD:operators.py` e o
  novo módulo.
- Identidade de objeto no reexport: `operators.<nome> is operator_sector_mapping.<nome>`
  para os 6.
- Sem import circular (`sector_aliases` não importa `operators`; deps lazy).
- Suite backend completa (banco de teste; nunca prod): verde.
