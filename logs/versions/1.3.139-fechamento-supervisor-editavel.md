# v1.3.139 — Fechamento: todos os campos editáveis (supervisor volta a ser editável)

**Data:** 2026-06-12
**Tipo:** Ajuste de produto (decisão do auditor) + testes
**Custo:** nenhum
**Branch:** `main`

## Decisão

Lucas: "deixe tudo selecionável; nós enviamos os dados na ordem correta para o
fechamento, porém se o auditor quiser alterar qualquer campo ele pode."

Ou seja, a planilha do Fechamento é **totalmente editável** pelo auditor. O
commit `74a2f10e` tinha tornado o **supervisor** read-only (fonte
única no cadastro). Esta versão reverte isso para o supervisor, **mantendo o
melhor dos dois mundos**:

- **Base (default)** = supervisor do cadastro vivo (`colaboradores.supervisor`
  / `db_supervisor`) — os dados chegam na ordem/valor corretos.
- **Override do auditor vence e persiste** por competência, igual aos outros
  12 campos. Para linhas que o auditor não tocar, segue o cadastro (supervisor
  removido reflete automaticamente — preserva a intenção do item 2/`74a2f10e`).

## Mudanças

`backend/core/fechamento_service.py`:
- `_format_registered_layout_row`: supervisor volta a respeitar `supervisor_override`.
- `_get_fechamento_rows_from_layout`: supervisor com precedência de override
  (`layout_supervisor_override` → `supervisor_override` → `db_supervisor` →
  `layout_supervisor`).
- `_get_fechamento_rows_legacy`: `supervisor_expr` e `ORDER BY` voltam a usar
  `f.supervisor_override` com fallback no cadastro.
- `save_fechamento_overrides` (layout + cadeia): supervisor volta a persistir
  via `_override_or_none(...)`.

`src/features/fechamento/components/FechamentoPage.tsx`:
- Select do supervisor reabilitado (sem `disabled`); `supervisorOptions` volta a
  incluir os supervisores presentes nas linhas; mantém a opção de fallback para
  valor fora da lista de ativos.
