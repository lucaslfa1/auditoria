# v1.3.114 — Guardrail de orçamento: teto diário, kill-switch e telemetria de custo

**Data:** 2026-06-11
**Tipo:** Feature — controle de custo (resposta estrutural ao estouro de orçamento Azure de jun/2026)
**Custo:** deploy backend + migration automática (`api_usage_daily`)
**Branch:** `handover`

## Contexto

O relatório de consumo (10/06) confirmou a causa do estouro: ciclos de 30 min
com engine `hybrid_dual` + 15 retries + classificação GPT de candidatas
descartadas. As mitigações v1.3.109–112 corrigiram os comportamentos, mas o
sistema continuava SEM teto estrutural — qualquer regressão de configuração
poderia estourar o orçamento de novo. Esta versão adiciona a garantia.

## Mudanças

### Nova tabela `api_usage_daily` (migration `m20260611_001`)
`(data, provider, categoria) -> chamadas`. Alimentada por UPSERT a cada
chamada paga. Serve de contador para os tetos e de telemetria por etapa.

### Novo módulo `backend/core/cost_guard.py`
- `record_call(provider, categoria)` — registrado IMEDIATAMENTE antes de cada
  request pago (tentativas contam; retries de rate-limit contam). Fail-open:
  falha de banco loga e segue.
- `budget_exceeded() -> Optional[str]` — motivo do bloqueio ou None:
  - `kill_switch_ativo` — env `COST_KILL_SWITCH` OU config `cost_kill_switch`
    (corta consumo pago via UPDATE no banco, sem redeploy);
  - `teto_auditorias_dia_atingido` — `COST_MAX_AUDITS_PER_DAY` (default **200**);
  - `teto_chamadas_llm_dia_atingido` — `COST_MAX_LLM_CALLS_PER_DAY` (default **1500**).
  Valor `<= 0` desativa o teto correspondente. Cache TTL 20s (não pesa no Neon).
- `record_audit_completed()` — 1 por item auditado com sucesso.
- `get_today_usage()` — resumo para a UI/status.

### Gates (verificação no INÍCIO de cada unidade de trabalho, nunca no meio)
- `automation_engine.run_automation_cycle`: ciclo nem começa
  (`status=budget_blocked`, visível no painel).
- `automation.audit_all_pending`: lote encerra graciosamente antes do próximo
  item; restantes continuam `ready_for_audit`; resultado ganha
  `budget_blocked_motivo` + mensagem.
- `huawei_sync._processar` (Fase 2): item retorna `budget_exceeded` e fica
  pendente — **nada é descartado**.
- `llm_triage.filtrar_ligacoes_com_llm`: retorna vazio → caller cai no
  `_triagem_fallback` determinístico (gratuito).

## Decisões de design
- **Fail-open**: indisponibilidade do banco não trava o pipeline; o teto é
  proteção de custo, não controle de segurança.
- **Contar tentativas** (não sucessos): tentativa falha também consome/risco.
- **Speech não conta no teto de LLM** (custo por hora de áudio é menor);
  fica visível na telemetria e é limitado indiretamente pelo teto de auditorias.

## Deploy (ação CI/CD)
1. Deploy backend normal (migration roda no prestart).
2. Opcional: setar envs `COST_MAX_*` no Cloud Run se quiser tetos diferentes
   dos defaults (1500 LLM / 200 auditorias por dia).
3. Validar: `GET /api/telefonia/sync/diagnostics` → bloco `custo_diario`.
