# v1.3.82 — Botão "Executar Manual" força re-download D-1 (bypass tentativas_esgotadas)

**Data:** 2026-05-27
**Tipo:** UX/Fix de fluxo
**Custo:** zero (só altera comportamento condicional)
**Bug relacionado:** BUG-032 (botão Manual Executar retornava "0 auditados" sem motivo claro)

## Contexto

Usuário (Lucas) reportou: "Botão iniciar automação manualmente (forçar) não funciona — ele informa que inicia e dá como 0 auditados concluindo o ciclo".

Investigação no Neon (`automation_cycle_runs`) e código mostrou que o sistema estava funcionando **como projetado**, mas o comportamento do botão divergia da expectativa do usuário:

**O que o botão fazia:**
1. `POST /api/automation/run-now` → `run_automation_cycle(source="manual_ui")`
2. Sync D-1: Tentava baixar áudios de 26/05, mas o estado `tentativas_esgotadas` (após `huawei_d1_max_retries=1` falhas do scheduler) **bloqueava** o download. Resultado: `pulados: [{date: 20260526, motivo: "tentativas_esgotadas"}]`.
3. Audit: Fila não tinha itens em `ready_for_audit` (apenas `pending`/`needs_manual_triage` aguardando triagem humana). Resultado: 0 auditados.

**Expectativa do usuário**: o botão deveria **forçar** re-execução do ciclo, ignorando os gates de retry que limitam o scheduler automático.

## Estratégia

Adicionar parâmetro `force=False` em `executar_d_minus_1_pipeline` que pula 3 gates de retry quando `True`:
- `tentativas_esgotadas` (retries esgotados)
- `antes_do_horario_programado` (horário de execução não chegou)
- `aguardando_intervalo_retry` (cooldown entre tentativas)

**Preservado** quando `force=True`:
- `ja_concluido` — não re-baixa dias que já fecharam com sucesso (evita trabalho desnecessário; o sistema deduplica por `input_hash` de qualquer forma)
- Todas as outras invariantes do pipeline (lock de execução, cancelamento, etc.)

Cloud scheduler **continua respeitando** todos os gates (passa `force=False` implícito). Apenas `source == "manual_ui"` aciona o bypass.

## Mudanças

### 1. `backend/core/huawei_d_minus_1.py`
- `_deve_executar(...)` ganha parâmetro `force: bool = False`. Quando `True`, os ifs de `tentativas_esgotadas`, `antes_do_horario_programado` e `aguardando_intervalo_retry` são pulados. `ja_concluido` continua barrando.
- `executar_d_minus_1_pipeline(...)` ganha parâmetro `force: bool = False` e propaga para `_deve_executar` no loop de lookback.

### 2. `backend/core/automation_engine.py`
- `run_automation_cycle(source=...)` passa `force=(source == "manual_ui")` ao `executar_d_minus_1_pipeline`. Sem outras mudanças — cloud_scheduler segue intocado.

### 3. `backend/tests/test_huawei_d_minus_1.py`
- `test_force_bypasses_tentativas_esgotadas` — confirma bypass de retries esgotados.
- `test_force_bypasses_antes_do_horario_programado` — confirma bypass do horário programado.
- `test_force_preserves_ja_concluido` — confirma que `force=True` NÃO re-roda dias completados.

## Resultado

- Botão "Executar Manual" agora consegue re-baixar D-1 mesmo após retries esgotados.
- Cloud scheduler mantém comportamento conservador (não força).
- Sem mudança em invariantes de segurança: o gate `ja_concluido` ainda protege contra re-trabalho redundante; o lock de ciclo continua impedindo execuções simultâneas.
