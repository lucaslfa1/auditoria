# v1.3.212 — Triagem de áudio puro: identifica o operador pela fala

**Data:** 2026-06-30
**Tipo:** Feature (triagem/classificação) — funcionar sem nome de arquivo
**Branch:** `main`

## Contexto

A triagem só acertava o setor quando o áudio vinha **da fila** (o nome do arquivo
da Huawei carrega operador/setor/alerta). Em **áudio puro** (upload solto), sem
esse nome, o sistema não conseguia identificar o setor e marcava "desconhecido".

Investigação confirmou o ponto exato: o casamento do operador com o cadastro exige
**nome + sobrenome** (`_is_strong_operator_name`, 2 tokens). No áudio a pessoa fala
só o **primeiro nome** ("aqui é a Valéria", "cadastro Opentech Dayane") → não casava.
Mas a informação ESTÁ na fala: o operador quase sempre se apresenta e diz o setor.

Importante: o mecanismo "operador reconhecido → banco força o setor oficial dele"
**já existia** (`enforce_operator_and_direction_guardrails`, força o setor do RH
inclusive sobre "desconhecido"). Faltava só **reconhecer o operador pela fala**.

## Mudança

Quando o operador NÃO foi resolvido pelo nome do arquivo nem pela IA (caso do áudio
puro), a triagem agora tenta identificá-lo pela transcrição:

1. Extrai o **primeiro nome** da apresentação ("aqui é a/o ___", "me chamo ___",
   "meu nome é ___", "<setor> Opentech ___").
2. Busca candidatos no cadastro por esse primeiro nome
   (`operators.listar_colaboradores_por_primeiro_nome`).
3. **Um** candidato → usa. **Vários** → desempata pelo **setor falado**
   ("logística", "cadastro", "sinistro"...). Continua ambíguo → **não mexe**
   (segue para revisão manual; nunca chuta — decisão do Lucas).
4. Achando o operador, o guardrail de RH existente puxa o setor oficial.

Tudo **à prova de erro 500**: `resolve_operator_from_speech` nunca levanta exceção
(qualquer falha → `None` → revisão manual), e o fluxo já é envolto por
`_safe_classify_audio`.
