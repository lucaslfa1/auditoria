# v1.3.185 — Fix: coleta manual "interrompida" agora fica visível (não trava em "Coletando…")

**Data:** 2026-06-18
**Tipo:** Correção de bug (confiabilidade/observabilidade) — item 2 do plano de force
**Custo:** nenhum
**Branch:** `main`

## Sintoma

"Puxar ligações agora" (Telefonia) terminava com frequência sem baixar nada: no
histórico, ~1/3 das coletas manuais ficavam `status='interrupted'`, `baixadas=0`,
e a UI podia ficar presa em "Coletando na Huawei…". O auditor achava que tinha
forçado a coleta com sucesso.

## Causa raiz

`POST /api/telefonia/sync/manual` dispara a coleta como **fire-and-forget**
(`asyncio.create_task` em [sync.py:120](../../backend/routers/telefonia_routes/sync.py))
e responde `accepted` na hora — a coleta roda **destacada do request**. No Cloud Run
(scale-to-zero + cpu-throttling) a CPU é estrangulada quando o tráfego cai, ou a
instância é reciclada num deploy → a task de segundo plano **morre sem chamar
`finalize`**. O run fica `finished_at=NULL`/`baixadas=0` e só era reconciliado para
`interrupted` no **próximo boot do app** (`reconcile_stale_telefonia_sync_runs`
chamada só em [main.py:225](../../backend/main.py)) — podia ser na manhã seguinte. E
`/sync/status` reporta estado **em memória** (`_LAST_SYNC`), que continua "running"
mesmo com a task morta, ou é forçado a "running" pelo `sync_lock` (TTL 30min).

## O que foi feito (mitigação/observabilidade — código only, sem infra)

1. **`/sync/status` honesto** ([sync.py](../../backend/routers/telefonia_routes/sync.py)):
   - reconcilia runs órfãos na leitura (`stale_after_seconds=600`);
   - cross-check no banco: se `_is_sync_running()` der True só pelo `sync_lock` mas
     não há run ativo fresco (`get_active_telefonia_sync_run` is None após o
     reconcile), o "running" é fantasma → reporta **`interrupted`** com mensagem
     clara ("Tente novamente"), em vez de spinner eterno.
2. **`/sync/history` honesto**: mesmo reconcile na leitura — o histórico mostra
   `interrupted` em ~10min em vez de só no próximo restart.

Ambos **auto-curam**: `finalize_telefonia_sync_run` sobrescreve por `run_id`, então
se a task ainda estiver viva (ex.: fase longa sem heartbeat) o resultado real
prevalece. Threshold de 600s evita falso-positivo em coleta lenta porém viva.

## O que NÃO foi feito (decisão consciente)

A **raiz** (fire-and-forget de tarefa longa em runtime serverless escala-a-zero) é
inerente ao botão manual. Eliminá-la de vez exige mudança arquitetural/infra (mover
a coleta para um Cloud Run Job dedicado, ou await em janela de CPU garantida, ou
`min-instances>=1`). **Não feito aqui** — o caminho confiável de coleta é o **cron
agendado** (awaitado dentro do request; ver v1.3.184), não o botão manual. Este fix
torna a falha do botão **visível e recuperável** (o auditor vê "interrompida" e
re-tenta, ou confia na coleta noturna).

## Validação

- `tests/backend/test_sync_status_interrupted.py` (novo, sem banco — mock de
  `_is_sync_running`/`get_active`/`reconcile`): **4 passed**. Cobre: memória
  "running" + task morta → interrupted; lock fantasma sem run ativo → interrupted;
  run ativo fresco → running; status terminal não é tocado.
- Módulo importa limpo (o import circular do teste foi só ordem de import —
  `routers.telefonia` antes do submódulo `sync`, como o app carrega).
- Suite completa só roda contra banco de teste (guard do conftest bloqueia prod);
  este fix foi verificado isolado com `DATABASE_URL` dummy não-prod.

Melhoria de observabilidade ainda pendente.
