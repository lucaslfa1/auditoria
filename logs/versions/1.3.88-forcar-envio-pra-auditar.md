# v1.3.88 — Auditor pode forçar envio à auditoria (sem "Corrigir antes" travando)

**Data:** 2026-05-27
**Tipo:** UX + comportamento — desbloqueio manual de itens "needs_manual_triage"
**Custo:** zero

## Contexto

Lucas (equipe de auditoria): "esse 'corrigir antes' travando o fluxo do auditor não é interessante. Deveria ser só pra automação e com opção do auditor poder forçar o envio."

A v1.3.87 já tinha relaxado o gate quando os motivos eram **apenas** de transcrição. O caso restante (motivos não-transcrição: classificação incompleta, direction guardrail, etc) ainda mostrava o botão "Corrigir antes" desabilitado com tooltip enganoso. Lucas pediu que o auditor SEMPRE possa enviar — o pipeline automático respeita o gate, mas o auditor humano sobrescreve.

## Decisão

- **Automação (ciclo)**: já não passa por aqui (só processa `ready_for_audit`).
- **Auditor humano** (UI): botão "Enviar para auditar" deixa de ficar trancado.
- 3 estados visuais:
  1. **Campos faltando** (setor/alerta/operador inválidos) → botão "Preencher campos" + click abre editor inline (não envia). É a única condição não forçável — falta dado pra pipeline rodar.
  2. **Status bloqueia + campos OK** (`needs_manual_triage` com motivos não-transcrição) → botão **âmbar "Forçar envio"** + `window.confirm` explicando o motivo. Auditor confirma e segue.
  3. **OK** → botão verde "Enviar para auditar" envia normal.
- `blocked_operator` continua bloqueado com mensagem específica (operador inexistente faria o pipeline falhar mesmo com `force`).

## Mudanças

### `backend/routers/telefonia.py`
- Novo `class AuditRecordingRequest(BaseModel): force: bool = False`.
- `POST /recordings/{hash}/audit` aceita body `{"force": true}`.
- Quando `force=true`:
  - Pula bloqueio de `needs_manual_triage` (independente dos motivos).
  - Pula `_raise_if_huawei_direction_blocked`.
  - Não pula `blocked_operator` (operador sem cadastro falha mesmo no pipeline).
  - Não pula `audited`/`monthly_capped` (audit ja existe / cota).
- Metadata: grava `audit_forced=True`, `audit_forced_by`, `audit_forced_at` pra rastreabilidade.
- Pydantic default puro (`= None`, não `Body(default=None)`) pra não quebrar tests que chamam a função diretamente.
- Import `from pydantic import BaseModel` adicionado.

### `src/features/classifier/components/RemoteTriageQueue.tsx`
- Removida `requiresManualCorrection` (booleano genérico).
- Adicionado `getMissingFields(item): string[]` — lista quais campos faltam.
- Adicionado `getForceableBlockReason(item): string | null` — razão legível quando há bloqueio forçável (ex: classificação incompleta).
- `handleSendToAudit({ force })`: se faltam campos abre editor; se forçável e não-force, confirma; envia com body `{force}`.
- Botão usa cor âmbar quando precisa forçar, verde quando normal. Label e tooltip refletem o estado.

### `src/features/telefonia/hooks/useTelefoniaSync.ts`
- Bugfix pré-existente: `SyncStatus` ganhou `message?: string`. Sem isso, `tsc -b` falhava em `SyncPanel.tsx:105` (`status?.message`) — bloqueando o build. Não estava no escopo, mas o build TS é gate do deploy via Cloud Build.
