# v1.3.93 — Fix +3h em huawei_begin_time vindo so do CSV manifesto

**Data:** 2026-05-28
**Tipo:** Bug fix critico — root cause do `audio_not_found` no cron D-1
**Custo:** zero (deploy backend)

## Contexto

O commit `7d2fac26` (2026-05-27) documentou que ~50% dos itens da fila chegavam
com `huawei_begin_time` +3h adiantado, causando ~22 `audio_not_found` em 7 dias.
A hipotese inicial foi que `is_manual=false` (cron automatico) era o discriminador,
e que `huawei_obs_client.py:122-143` ou `huawei_discovery.py:21-35` introduziam
o offset por falta de `.astimezone(utc)`. Phase 3 do fix estava pendente.

## Root cause

O discriminador real e o `huawei_source` no metadata, nao `is_manual`:

| huawei_source                | diff vs call_id_epoch |
|------------------------------|-----------------------|
| `obs_contact_record+vdn`     | 0h                    |
| `vdn`                        | 0h                    |
| `obs_contact_record` (so CSV)| **+3h**               |

Dados de prod confirmaram em 5/5 amostras `obs_contact_record`-only: tanto
`huawei_begin_time` quanto `huawei_end_time` ficam +3h adiantados, com diferenca
`end - begin` igual a duracao real da chamada. Manual e auto-cron tinham
proporcao similar de bug — `is_manual` era pista falsa.

**Causa raiz**: o CSV manifesto do OBS da Huawei envia `beginTime` e `endTime`
como string ISO `"YYYY-MM-DD HH:MM:SS"` em UTC (sem fuso explicito).
`HuaweiDiscoveryService._coerce_huawei_time_ms` em `huawei_discovery.py:31`
assumia BRT (`replace(tzinfo=ZoneInfo("America/Sao_Paulo"))`), entao ao converter
para epoch UTC somava +3h ao valor real.

Itens que tambem aparecem na API VDN (`obs_contact_record+vdn` ou `vdn`) ficam
corretos porque a API retorna `beginTime` como numero (epoch UTC) e ganha no
`merge_interacoes` (primeiro a chegar com valor nao-vazio fica).

## Validacao da tese (Phase 3)

Simulacao Python com call_id epoch real (`1779798227`):

```
input ISO UTC "2026-05-26 12:23:47" -> coerce as BRT -> 1779809027000 (+3.00h)
input ISO BRT "2026-05-26 09:23:47" -> coerce as BRT -> 1779798227000 (0.00h)
stored value em prod                                  1779809093000 (+3.02h)
```

Reproducao bit-a-bit do bug observado (diferenca de 66ms = call comecou ~1min
depois do "head epoch" do call_id).

## Mudancas

### `backend/core/huawei_discovery.py`

`_coerce_huawei_time_ms` (linhas 22-35): troca `ZoneInfo("America/Sao_Paulo")`
por `timezone.utc` no branch de parse de string ISO.

```python
# antes
dt = datetime.strptime(text, fmt).replace(tzinfo=ZoneInfo("America/Sao_Paulo"))
return int(dt.astimezone(timezone.utc).timestamp() * 1000)

# depois
dt = datetime.strptime(text, fmt).replace(tzinfo=timezone.utc)
return int(dt.timestamp() * 1000)
```

Branch numerico (`_coerce_epoch_millis`) intocado — API VDN ja retornava UTC
correto e nao precisa de conversao.

### `tests/backend/test_huawei_sync.py`

Dois testes de regressao adicionados:

- `test_coerce_huawei_time_ms_iso_string_assumes_utc` — input ISO `"2026-05-26 12:23:47"`
  deve retornar `1779798227000` (UTC epoch direto).
- `test_manifest_row_iso_string_preserves_call_id_epoch` — fluxo via
  `_manifest_row_to_interacao` preserva o epoch do call_id, sem +3h drift.

Ambos falham no codigo antigo (`1779809027000 != 1779798227000`) e passam
no novo.

## Defesa em profundidade

### `backend/core/huawei_obs_client.py`

`_coerce_to_epoch_ms` (linhas 122-143): mesma troca BRT → UTC, evita
desalinhamento de `_candidate_dates` (busca YYYYMMDD no OBS) caso um
`huawei_begin_time` chegue como string ISO em algum fallback. Branch numerico
intocado.

`tests/backend/test_huawei_obs_client.py`: `test_coerce_to_epoch_ms_iso_string_assumes_utc`
adicionado (mesmo input `"2026-05-26 12:23:47"`, mesmo epoch UTC esperado).

## Nao tocado (escopo controlado)

- `backend/core/automation_rules.py:142` (`_coerce_time_millis`) — input ja
  passou pela conversao em `_manifest_row_to_interacao` antes, entao branch ISO
  nao roda na pratica. Sem bug observavel.
- `backend/routers/telefonia.py:615` (`_parse_iso_to_ms`) — INTENCIONAL para
  `datetime-local` do navegador que chega sem TZ; mantido em BRT.

## Dead code residual removido

`backend/core/huawei_sync.py:454-460` continha um trecho orfao:

```python
for fmt in ("%Y-%m-%d %H:%M:%S", "%Y/%m/%d %H:%M:%S"):
    try:
        dt = datetime.strptime(text, fmt).replace(tzinfo=ZoneInfo("America/Sao_Paulo"))
        return int(dt.astimezone(timezone.utc).timestamp() * 1000)
    except ValueError:
        continue
return None
```

Era resto da funcao `_coerce_huawei_time_ms` que existia em `huawei_sync.py` no
commit `32a5ebb8c` (2026-04-26) e foi movida para `huawei_discovery.py` num
refator posterior. As 7 ultimas linhas do corpo ficaram orfas, sem cabecalho.

O commit `552e4f65` (v1.3.89, pause/resume) colou a definicao de
`_wait_if_paused` imediatamente acima desse trecho, fazendo parecer que faziam
parte da mesma funcao. Como a variavel `text` nao existia no escopo de
`_wait_if_paused`, o trecho cresceria com `NameError` se executado — escapava
em runtime porque o caminho `pause -> resume` quase nunca era exercitado em
producao (so na nova UI de pause/resume da v1.3.89, ainda sem fluxo testado de
pausa real).

Verificacao de que era dead code: as 3 chamadas de `_coerce_huawei_time_ms` em
`huawei_sync.py` (linhas 592, 593, 686 na arvore atual) ja usam
`HuaweiDiscoveryService._coerce_huawei_time_ms(...)` (qualificado), nao a
versao local. Nenhum caller esperando a definicao local.

Acao: removido. `_wait_if_paused` termina corretamente em `logger.info("Sync Huawei: retomado pelo usuario.")`.
