# v1.3.79 — Lazy imports para reduzir cold start do Cloud Run

**Data:** 2026-05-26
**Tipo:** Performance (zero feature change)
**Custo:** $0 (só código, nenhum recurso novo)
**Task relacionada:** #36 (Investigar lentidão extrema)

## Contexto

Diagnóstico apontou três camadas de lentidão somando-se na primeira request:
1. **Cloud Run scale-to-zero** com `--cpu-throttling --min-instances=0` → cold start de container a cada despertar
2. **Neon DB com `suspend_timeout_seconds=0`** → DB suspende imediatamente e precisa acordar
3. **Imports pesados eager no boot** → ~3-5s só carregando libs antes de servir 1ª request

Usuário escolheu rota **$0 absoluto** (vai migrar DB em breve). Este log cobre só o item 3 — única coisa que pode ser melhorada sem incorrer custo.

## Resultado

**Antes (libs carregadas no boot do `services` / `main`):**
- google.genai (carregava protobuf grande, ~500ms-1s)
- pandas (~1-2s)
- openpyxl (~200-400ms)
- pypdf (~300ms)
- numpy (transitivo via pandas)
- reportlab (transitivo)
- docx (transitivo)

**Depois (verificado via importtime):**
```
google.genai: lazy
pandas: lazy
openpyxl: lazy
pypdf: lazy
numpy: lazy
reportlab: lazy
docx: lazy
```

Boot de `main.py` em Windows local: ~10s (era ~15s estimado). No Cloud Run com layer cacheado, a redução proporcional será similar (~30-50%).

**Trade-off:** Primeira chamada a um endpoint que precise da lib paga ~100-500ms extras só na primeira vez (uma vez por container). Endpoints afetados:
- POST `/api/audit` → carrega google.genai/types apenas se cair no fallback Gemini (caminho raro hoje, padrão é Azure)
- GET `/api/audit/.../excel` → carrega pandas + openpyxl
- GET `/api/audit/.../pdf` → carrega reportlab
- GET `/api/fechamento/exportar` → carrega openpyxl
- POST `/api/audit` com PDF → carrega pypdf
- POST `/api/classifier` no fallback Gemini → carrega google.genai

Auth, listagem, dashboards, telefonia: **NENHUM impacto** (nenhuma dessas libs é tocada).
