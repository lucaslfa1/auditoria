## BUGs corrigidos (em ordem aproximada de gravidade)

### Criticos (paralisavam operacao)

#### BUG-020 -- 422 silencioso em PUT /api/audit/draft (auto-save)
- **Causa:** `apiFetchJson` em `src/shared/lib/apiClient.ts` enviava
  `body: JSON.stringify(...)` sem `Content-Type: application/json`. FastAPI
  rejeitava como `dict_type` 422. Frontend silenciava o erro
  (`.catch(console.error)`).
- **Sintoma:** auditores reportavam travamento e bloqueio ao salvar.
- **Impacto:** dezenas de chamadas espalhadas (auto-save audit, admin
  criteria, feedback IA, triagem manual).
- **Fix:** `apiFetch` adiciona `Content-Type: application/json`
  automaticamente quando body comeca com `{`, `[` ou `"` e nao ha header
  setado. Preserva FormData/Blob (uploads multipart continuam funcionando).
- **Commit:** `2a284714` (deploy 00842/00843)

#### BUG-024 -- ROUND(double precision, int) inexistente no PostgreSQL
- **Causa:** PostgreSQL nao tem `ROUND(double, integer)`, apenas
  `ROUND(numeric, integer)`. Quando AVG retornava double, falhava com
  `psycopg2.errors.UndefinedFunction`.
- **Sintoma:** GET `/api/operadores/`, `/api/supervisores/*`,
  `/api/operadores/{id}/auditorias`, `/performance` e exports PowerBI
  retornavam HTTP 500 silencioso. Frontend mostrava lista vazia.
  Auditora reportou: "Victor e Poliana cadastrei mas nao aparecem"
  (eles existiam mas a lista inteira falhava).
- **Fix:** cast explicito `::numeric` antes do ROUND em 3 arquivos:
  - `backend/routers/operadores.py`
  - `backend/routers/supervisores.py`
  - `backend/repositories/report_powerbi.py`
  (`backend/repositories/analytics_quality.py` ja tinha o cast correto e
  serviu de referencia)
- **Commit:** `6d33d2e1` (deploy 00844)

#### BUG-019 -- Fallback silencioso para localhost em `get_database_url`
- **Causa:** `backend/db/connection.py:79-89` retornava
  `postgresql://postgres:postgres@localhost:5432/auditoria` quando
  `DATABASE_URL` estava vazia. Sem log, sem alerta.
- **Contexto operacional:** `backend/.env:72` tinha typo
  `SENTRY_RELEASE=DATABASE_URL=postgresql://...` (a string da connection
  estava GRUDADA no SENTRY_RELEASE), e `.env` raiz apontava para banco
  antigo. Em desenvolvimento local Claude conectou silenciosamente no
  banco errado por meia hora antes de descobrir.
- **Fix:** raise `RuntimeError` em producao (`ENVIRONMENT=production`);
  log WARNING explicito em dev antes do fallback.
- **Commit:** `2a284714`

#### BUG-023 -- Cloud Run em prod gravava no banco errado
- **Causa:** env var DATABASE_URL no Cloud Run apontava para
  `ep-falling-hall-ac2t9rln-pooler` (banco antigo). Auditorias do dia
  iam para la, mas a UI lia do `ep-aged-river-acr5e219-pooler`
  (canonico). Por isso "auditoria nao aparece na lista".
- **Resolucao:**
  - Criada branch backup do aged-river no Neon
    (`backup-pre-migration-falling-hall-2026-05-26-13h`,
    `br-billowing-sound-acptp03x`).
  - Migrados 67 audits + 13 arquivos_salvos + 218 transcript_candidates
    + 19 colaboradores (depois revertidos) de falling-hall ->
    aged-river preservando IDs (zero overlap).
  - Sequences sincronizadas.
  - DATABASE_URL no Cloud Run atualizada (pelo CI/CD) para aged-river.

### Regras de negocio refinadas

#### BUG-025 -- Dedup de id_huawei priorizando ATIVO+auditavel+recente
- **Causa:** `list_colaboradores` deduplicava por `id_huawei` usando
  `ORDER BY nome`. Em handover de ramal (operador sai, outro assume),
  o titular ATUAL desaparecia se o nome do anterior viesse primeiro
  no alfabeto.
- **Sintoma:** Victor (1164) e Poliana (1163) sumiam porque Jucenir
  (1050) e Fernanda (1061) vinham antes alfabeticamente.
- **Fix:**
  ```sql
  ORDER BY
    CASE WHEN UPPER(status) = 'ATIVO' THEN 0 ELSE 1 END,
    CASE WHEN COALESCE(auditavel, 1) = 1 THEN 0 ELSE 1 END,
    atualizado_em DESC NULLS LAST,
    nome
  ```
- **Commit:** `426e66cd` (deploy 00845)

#### BUG-026 -- auditavel=0 nao deve virar INATIVO no fechamento
- **Causa:** `_resolve_fechamento_status` em
  `core/fechamento_service.py:251` forcava `status="INATIVO"` quando
  `auditavel=False`, escondendo do fechamento qualquer operador
  ATIVO+nao-auditavel.
- **Sintoma:** auditora Fatima reportou: "a Fernanda precisa ficar
  para o fechamento mesmo nao sendo auditada".
- **Fix:** `_resolve_fechamento_status` agora respeita apenas o status
  real (`colaboradores.status`). A flag `auditavel` so vale para o
  pipeline da IA. Teste `test_nao_auditavel_is_inativo_in_fechamento`
  reescrito para refletir nova regra.
- **Commit:** `1d6d6c9e`

### Cloud Run perf (BUG-036)
- Config atual: cpu=1, concurrency=80, min-instances=0, cpu-throttling=true.
- Sintoma observado nos logs: favicon de 122s, audit de 250s, lookup
  operadores de 80s. Indicio de thrashing + cold start severo.
- Fix recomendado (mas adiado por custo +$20-30/mes):
  ```bash
  gcloud run services update auditoria --region=southamerica-east1 \
    --concurrency=8 --cpu=2 --memory=4Gi \
    --min-instances=1 --no-cpu-throttling
  ```

### AZURE_TRANSCRIPTION_ENGINE
- Em prod: `hybrid_dual`. Memoria do projeto (`project_transcription_pipeline_2026_05.md`)
  + commit `98f7bd3` indicam que o default oficial mudou para `fast`
  em 2026-05-20. Dados objetivos do teste confirmam que `fast` ganha
  na maioria dos audios (score deterministico maior).
- Comando para CI/CD executar quando decidir:
  ```bash
  gcloud run services update auditoria --region=southamerica-east1 \
    --update-env-vars AZURE_TRANSCRIPTION_ENGINE=fast
  ```
- Adiado: o candidate selector ja roda os 4 providers e escolhe o
  melhor, entao o impacto pratico de mudar o engine e apenas economia
  de tempo/custo (evita rodar `hybrid_dual` extra alem dos 4).
