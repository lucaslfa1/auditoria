## Objetivo
Dois fixes independentes detectados na mesma sessao de debug pos-1.3.65:

1. **Edicao de operador retornando HTTP 500** em todo PUT `/api/admin/colaboradores/{id}`.
2. **Guardrail de setor↔alerta nao disparando** quando o setor cru do RH (ex: "BASE PR - AZUL") nao bate diretamente com uma chave canonica do catalogo.

## Sintomas

### Bug 1 — 500 ao editar
Operadora `LIBIA CONCEICAO OLIVEIRA CORDEIRO` (matricula 11318) tentou ser editada via tela de Operadores. Retorno: `Erro no Servidor — Request failed (500)`. Reproduzia para qualquer colaborador.

### Bug 2 — Alerta de outro setor
Mesma operadora teve uma ligacao triada como `LOGISTICA-PARADA`, mas seu setor canonico (resolvido por `_operator_sector_id`) e `bas` ("BASE PR - AZUL" -> startswith("bas")). O alerta `LOGISTICA-PARADA` nao pertence ao catalogo BAS, mas a guarda nao corrigiu — gravou no item da fila um par incoerente: `setor=bas` + `alert=LOGISTICA-PARADA`.

## Causas

### Bug 1
O refactor que separou `database.py` em `repositories/operators.py` deixou um descasamento de keyword: o router (`backend/routers/admin.py:243-244`) chama
```python
operators.update_colaborador(database.get_connection,
    colaborador_id=operador_id, ...)
```
mas a funcao em `backend/repositories/operators.py:1042-1059` declarava o parametro como `operador_id: int`. Resultado:
```
TypeError: update_colaborador() got an unexpected keyword argument 'colaborador_id'
```
FastAPI converte para 500. A funcao tinha um unico call site, e este ja usava `colaborador_id=`.

### Bug 2
Em `backend/core/huawei_sync.py:1117,1155-1163`, `_classificar_audio_huawei` passava o **texto cru** do setor do RH para `enforce_operator_and_direction_guardrails(db_sector=...)`:

```python
operator_sector = str(operador.get("setor") or "").strip().lower()  # "base pr - azul"
...
final_sector = operator_sector or resolved_operator.db_sector or ""
if final_sector:
    classification = enforce_operator_and_direction_guardrails(
        ..., db_sector=final_sector, ...)
```

Dentro da guarda (`backend/classification.py:910,924`), `_resolve_db_sector_alias("base pr - azul")` devolve `None` (nao esta em `sector_alias_map` nem em `audit_criteria_catalog`). Com `resolved_db_sector=None`, o `if resolved_db_sector and resolved_db_sector != sector_id` nao dispara, e a guarda inteira e pulada — o alerta da IA permanece, mesmo de outro setor.

O setor canonico ja era resolvido em `_operator_sector_id(operador)` (mesmo arquivo, linha 308): consulta `database.map_db_sector_to_classification_sector` e devolve "bas" para "BASE PR - AZUL". A guarda so precisava receber esse valor.

## Mudancas

### `backend/repositories/operators.py`
- Renomeado parametro `operador_id` -> `colaborador_id` em `update_colaborador` (linhas 1044 e 1094, ambas as ocorrencias dentro da funcao). Alinha com o nome usado pelo unico call site.

### `backend/core/huawei_sync.py`
- `_classificar_audio_huawei` agora resolve o setor canonico via `_operator_sector_id(operador)` antes de passar para `enforce_operator_and_direction_guardrails`. Mantida a fallback chain anterior para casos em que o canonico nao puder ser determinado.

## Comportamento Pos-Fix

### Bug 1
PUT `/api/admin/colaboradores/{id}` passa a executar o UPDATE corretamente. Edicao manual de qualquer colaborador volta a funcionar.

### Bug 2
Para uma operadora com `setor="BASE PR - AZUL"` e IA classificando como `LOGISTICA-PARADA`:
1. Guarda recebe `db_sector="bas"` (canonico).
2. `_resolve_db_sector_alias("bas")` -> "bas" (ja canonico).
3. `bas != "logistica"` -> override de setor para "bas".
4. `LOGISTICA-PARADA` nao esta no catalogo BAS -> tenta hint do filename, depois keyword "PARADA", e por fim cai em `alert_id="desconhecido"` com reason `alerta_incompativel_com_setor_operador`.

Resultado: nunca mais pares `setor=X + alert=Y de outro setor` para operadores com setor RH ja mapeado canonicamente.

## Riscos
- Bug 1: zero. So existe um chamador, e o nome ja batia.
- Bug 2: a guarda passa a ativar para mais operadores (qualquer um cuja `setor` do RH bate via `map_db_sector_to_classification_sector`). Isso pode aumentar o volume de itens com `review_reasons=["verificar_setor"]` ou `alerta_incompativel_com_setor_operador` na fila — mas e exatamente o sinal que faltava antes (alertas silenciosamente errados).
