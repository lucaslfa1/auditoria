# v1.3.110 - Cache do health-snapshot + polling adaptativo (corte de custo Neon)

**Data:** 2026-06-10
**Tipo:** Custo / Performance (Neon)
**Status:** implementado, `tsc -b` limpo, 13/13 testes passam, commitado e enviado para `main` (deploy automático via Actions)

## Contexto

Ainda na resposta à emergência de custo, mas numa frente **diferente** da
v1.3.109: aqui é custo de **banco (Neon)**, não de Azure. As telas
Telefonia/Settings fazem **polling** do status da automação. Cada poll
re-consultava agregados de fila/sync/ciclo no Neon **mesmo com a automação
ociosa**, gerando wakeups de compute e transferência de rede desnecessários.

Origem do refactor: apareceu no working tree durante a sessão; verificado e finalizado.

## Mudança

**`backend/core/automation_engine.py`:**
- Novo `_load_health_snapshot_cached()` que envolve `_load_health_snapshot()`
  com **TTL** (`AUTOMATION_HEALTH_SNAPSHOT_TTL_SECONDS`, default **30s**) via
  cache em memória (`_health_snapshot_cache`). `TTL=0` desativa o cache.
- **Ligado** em `get_engine_status()` (linha ~1662): a montagem do
  `health_report` passa a usar o snapshot cacheado em vez de bater no DB a cada
  chamada.

**`src/features/telefonia/hooks/useTelefoniaSync.ts`:**
- Polling **adaptativo**: 8s quando `is_running`/`is_cycle_running`; **60s em
  repouso**. Reduz chamadas de status (e, por tabela, consultas ao Neon) quando
  não há nada acontecendo.

**`tests/backend/test_automation_control_state.py`:**
- Mock de `_reconcile_stale_running_cycles` no teste de lock ocupado, deixando-o
  hermético (sem tocar o DB durante o skip do ciclo).

## Arquivos alterados
